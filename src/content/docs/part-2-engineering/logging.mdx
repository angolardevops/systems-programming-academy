When production misbehaves at 3 a.m., your logs are the crime scene. `print("here
2")` will not save you; what saves you is **structured** logging — one
machine-parseable line per event, with key-value fields you can filter
(`user_id=42`, `request_id=abc-123`), severity **levels** to separate signal from
noise, and **context** that ties all the lines of one request together. This lesson
builds the same disciplined logger in three languages — and, just as importantly,
shows how to **test** what your code logs.

## The four properties of production logging

1. **Structured** — events are key-value data (`{"level":"INFO","msg":"user logged
   in","user_id":42}`), not prose. Machines aggregate them; humans query them.
2. **Levelled** — `DEBUG` < `INFO` < `WARN` < `ERROR`. A threshold filters noise
   without touching call sites.
3. **Testable** — the sink (where lines go) is injected, so a test captures output
   in a buffer and asserts on it — exactly like injecting a repository.
4. **Contextual** — bind fields like `request_id` once; every subsequent line
   carries them, so one grep reconstructs a request's whole story.

<Mermaid title="One request, one context: bound fields tie the lines together" code={`flowchart LR

  B --> L2["WARN slow query ms=250"]

## A structured, injectable logger, three ways

Go's answer ships in the standard library (`log/slog`); Python's `logging` gets a
small JSON formatter; Rust we hand-roll (production Rust uses `tracing`):

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="structlog_demo.py"
class JsonFormatter(logging.Formatter):
    """One JSON line per record: level, msg, plus any extra= fields."""

    def format(self, record: logging.LogRecord) -> str:
        payload: dict[str, Any] = {
            "level": record.levelname,
            "msg": record.getMessage(),
        }
        for key, value in record.__dict__.items():
            if key not in _STANDARD_ATTRS:
                payload[key] = value  # fields passed via extra=
        return json.dumps(payload)


def new_logger(name: str, sink: IO[str], level: int = logging.INFO) -> logging.Logger:
    logger = logging.getLogger(name)
    logger.setLevel(level)
    logger.handlers.clear()
    logger.propagate = False
    handler = logging.StreamHandler(sink)  # the sink is injected
    handler.setFormatter(JsonFormatter())
    logger.addHandler(handler)
    return logger
```

```go title="logging.go"
// log/slog is the stdlib answer: levelled, structured, pluggable handlers.
func New(sink io.Writer, level slog.Level) *slog.Logger {
	handler := slog.NewJSONHandler(sink, &slog.HandlerOptions{
		Level: level,
		ReplaceAttr: func(groups []string, a slog.Attr) slog.Attr {
			if a.Key == slog.TimeKey {
				return slog.Attr{} // drop time for reproducible tests
			}
			return a
		},
	})
	return slog.New(handler)
}

// Bind a context field: every line through the returned logger carries it.
func WithRequestID(logger *slog.Logger, id string) *slog.Logger {
	return logger.With("request_id", id)
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
// Generic over any Write sink — stderr in prod, Vec<u8> in tests.
pub struct Logger<W: Write> {
    sink: W,
    min_level: Level,
    context: Vec<(String, String)>, // bound fields, on every line
}

impl<W: Write> Logger<W> {
    pub fn new(sink: W, min_level: Level) -> Self {
        Logger { sink, min_level, context: Vec::new() }
    }

    // Bind a context field (builder style).
    pub fn with_field(mut self, key: &str, value: &str) -> Self {
        self.context.push((key.to_string(), value.to_string()));
        self
    }

    pub fn log(&mut self, level: Level, msg: &str, fields: &[(&str, &str)]) {
        if level < self.min_level {
            return; // filtered out
        }
        let mut line = format!("{{\"level\":\"{level}\",\"msg\":\"{msg}\"");
        for (k, v) in self.context.iter().map(|(k, v)| (k.as_str(), v.as_str())).chain(fields.iter().copied()) {
            line.push_str(&format!(",\"{k}\":\"{v}\""));
        }
        line.push('}');
        let _ = writeln!(self.sink, "{line}");
    }
}
```

## Testing what your code logs

Logs are behaviour — a `WARN` on a failed login is part of the contract. Because
the sink is injected, a test captures output and asserts on it:

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="test_structlog_demo.py"
def test_context_field_on_every_line(self):
    sink = io.StringIO()
    logger = with_request_id(new_logger("t3", sink), "abc-123")

    logger.info("start")
    logger.warning("slow query", extra={"ms": 250})

    for line in lines_of(sink):
        self.assertEqual(line["request_id"], "abc-123")
```
</TabItem>
<TabItem label="Go">
```go title="logging_test.go"
func TestContextFieldOnEveryLine(t *testing.T) {
	var buf bytes.Buffer
	logger := WithRequestID(New(&buf, slog.LevelInfo), "abc-123")

	logger.Info("start")
	logger.Warn("slow query", "ms", 250)

	for _, line := range strings.Split(strings.TrimSpace(buf.String()), "\n") {
		if m := decode(t, line); m["request_id"] != "abc-123" {
			t.Errorf("line missing request_id: %v", m)
		}
	}
}
```

<TabItem label="Rust">

```rust title="src/lib.rs (tests)"
fn context_fields_appear_on_every_line() {
    let mut log = Logger::new(Vec::new(), Level::Info).with_field("request_id", "abc-123");
    log.info("start", &[]);
    log.warn("slow query", &[("ms", "250")]);
```

## The three pillars of observability

Logging is one pillar of three. **Logs** answer "what happened?" (discrete events).
**Metrics** answer "how much / how fast?" (aggregable numbers: request counts,
latency histograms — cheap to store, ideal for dashboards and alerts). **Traces**
answer "where did the time go?" (one request's path across services, each step a
timed span). The `request_id` we bound is the seed of tracing — promote it to a
trace ID propagated across service calls and you have distributed tracing
(OpenTelemetry standardises this for all three languages).

| Pillar | Question | Shape | Typical tools |
| --- | --- | --- | --- |
| Logs | What happened? | events with fields | slog/tracing/logging → Loki, ELK |
| Metrics | How much? How fast? | counters, gauges, histograms | Prometheus + Grafana |
| Traces | Where did time go? | spans in a request tree | OpenTelemetry, Jaeger |

## Common mistakes & debugging

### 1. Logging secrets or PII

The config lesson's redaction rule applies with force here: log lines are copied
into aggregators, retained for months, and read by many eyes. Never log passwords,
tokens, or full personal data; log opaque IDs instead.

### 2. Wrong levels everywhere

If everything is `ERROR`, alerts are meaningless; if real failures are `INFO`,
nobody wakes up. Rule of thumb: `ERROR` = a human should look; `WARN` = degraded
but self-healing; `INFO` = state changes worth an audit trail; `DEBUG` =
development detail, off in production.

### 3. Logging in a hot loop

A log line per item in a million-item loop costs serialization, I/O, and storage —
and buries the signal. Log the summary (`processed=1000000 failed=3`), and use
metrics for rates.

Write the test proving `handle_login(..., ok=false)` emits a `WARN`-level line with

**Solution (Python):**

```python
def test_failed_login_logged_as_warning(self):

    logger = new_logger("t4", sink)

    handle_login(logger, 7, ok=False)

    (line,) = lines_of(sink)
```

This is a behavioural contract: operations dashboards alert on this line.

<Interview question="Exercise 2 — Add level filtering to a noisy component.">

Show how to run one component at WARN while the rest of the app logs INFO. (Pick
your language.)

**Solution (Go):**

```go
appLogger := New(os.Stderr, slog.LevelInfo)       // app-wide: INFO+
dbLogger := New(os.Stderr, slog.LevelWarn)        // chatty component: WARN+
dbLogger = dbLogger.With("component", "db")
```

Two handlers with different thresholds; the `component` field keeps the source
identifiable in the merged stream.

</Interview>

<Interview question="Exercise 3 — Explain the injected sink.">

In one sentence: why does the logger take its output writer as a parameter?

**Answer:** For the same reason the repository and the environment were injected —
tests can substitute an in-memory buffer and assert exactly what was logged, while
production passes stderr or a file, without changing any call sites.

</Interview>

## Best practices vs anti-patterns

| ✅ Best practice | ❌ Anti-pattern |
| --- | --- |
| Structured key-value fields | Variables interpolated into prose messages |
| Bind `request_id` context once | Copy-pasting the ID into every message |
| Test important log lines | Treating logs as unverifiable side effects |
| Correct levels; DEBUG off in prod | Everything at ERROR (or everything at INFO) |
| Log IDs, never secrets/PII | Tokens and emails in the log stream |
| Summaries + metrics for hot paths | A log line per iteration of a hot loop |

## Interview questions

<Interview question="What is structured logging and why does it beat printf-style logging?">

Structured logging emits each event as data — typically one JSON object per line
with a stable message plus key-value fields — instead of formatting values into a
prose string. It wins because log aggregators index the fields: you can query
`level=ERROR AND user_id=42`, graph rates, and alert, all without regex-parsing
free text. It also keeps messages stable (better grouping) and makes logs testable
as data.

</Interview>

<Interview question="What are the three pillars of observability?">

Logs, metrics, and traces. Logs record discrete events with context — best for
"what happened here?". Metrics are cheap aggregable numbers (counters, gauges,
histograms) — best for dashboards, trends, and alerting. Traces follow a single
request across components and services as timed spans — best for finding where
latency lives. They complement each other; a `request_id`/trace ID links a metric
anomaly to its traces and those traces to their logs.

</Interview>

<Interview question="How do you test logging?">

Inject the sink: construct the logger over an in-memory buffer (`Vec<u8>`,
`bytes.Buffer`, `StringIO`) in tests, run the code, then parse the captured lines
and assert on level, message, and fields. This treats important log lines as part
of the observable contract — a failed login *must* produce a WARN with the user id
— so refactors can't silently destroy the operational visibility your on-call
depends on.

</Interview>

<Interview question="When should something be a metric instead of a log line?">

When you care about the aggregate, not the individual event — request rates, error
percentages, latency percentiles, queue depths. Metrics cost O(1) storage per
series regardless of traffic, are pre-aggregated for fast dashboards, and drive
alerting cleanly. Logs are for events you'll need to inspect individually, with
context. The classic smell is deriving a rate by counting log lines — that's a
counter metric wearing a log costume.

</Interview>

## Challenge

Take a service you own and reconstruct one real request from its logs alone: can
you find every line it produced, in order, with nothing missing and nothing from
other requests mixed in? If not — and without bound `request_id` context, you
usually can't — retrofit this lesson's pattern: structured lines, a request-scoped
context field, correct levels. Then break something on purpose and time how long
the logs take to tell you *what* and *why*. That time is your real MTTR, and this
discipline is what shrinks it.
