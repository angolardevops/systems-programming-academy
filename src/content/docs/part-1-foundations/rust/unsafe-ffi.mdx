Every abstraction bottoms out somewhere. Rust's safety guarantees are built on a
small amount of code that the compiler *cannot* verify — talking to hardware, to
the operating system, to C libraries, or expressing a pattern the borrow checker
is too conservative to allow. `unsafe` is the keyword that says "I've checked this
by hand." Used well, it is rare, tiny, and wrapped in a safe API. This lesson
demystifies it and shows the two directions of Foreign Function Interface (FFI).

<Aside type="caution" title="This is expert-level, and rarely needed">
  Most Rust programmers write little or no `unsafe`. Reach for it only when you
  must, keep it minimal, document why it's sound, and — where possible — verify
  it with [Miri](https://github.com/rust-lang/miri). The goal is a safe API on
  top; callers should never need `unsafe` themselves.
</Aside>

All code is from the tested crate
[`examples/rust-unsafe/`](https://github.com/angolardevops/systems-programming-academy/tree/main/examples/rust-unsafe)
(7 passing tests, clean `cargo clippy`). Every snippet is **complete**.

## What `unsafe` is — and is not

A common myth: "`unsafe` turns off the borrow checker." It does not. Inside an
`unsafe` block the borrow checker, type checker, and every other rule still apply.
`unsafe` unlocks exactly **five** extra abilities:

1. Dereference a raw pointer (`*const T`, `*mut T`).
2. Call an `unsafe` function (including all FFI functions).
3. Implement an `unsafe` trait (e.g. `Send`/`Sync` by hand).
4. Access or modify a mutable `static`.
5. Access fields of a `union`.

In exchange, *you* promise to uphold the invariants the compiler can no longer
check — chiefly that references are valid and never alias illegally.

## Raw pointers

You can *create* raw pointers in safe code; only *dereferencing* them needs
`unsafe`, because the compiler can't prove they point at valid, live data.

```rust title="src/lib.rs"
pub fn raw_pointer_roundtrip(value: i32) -> i32 {
    let r = &value;
    let raw: *const i32 = r; // making a raw pointer is safe

    // SAFETY: `raw` came from a live reference to `value`, still in scope, so it
    // is valid and aligned.
    unsafe { *raw } // dereferencing it is unsafe
}
```

<Aside type="tip" title="Write a SAFETY comment for every unsafe block">
  The convention — used throughout the standard library — is a `// SAFETY:`
  comment above each `unsafe` block explaining *why* the invariants hold. If you
  can't write that justification, the code isn't sound. `clippy` can even require
  it (`undocumented_unsafe_blocks`).
</Aside>

## A safe abstraction over an unsafe core

The canonical example: `split_at_mut` returns two mutable slices into one slice.
Safe Rust can't express this — the borrow checker can't prove the halves don't
overlap — so the standard library implements it with a few lines of `unsafe`,
behind a completely safe signature.

```rust title="src/lib.rs"
use std::slice;

pub fn split_at_mut<T>(slice: &mut [T], mid: usize) -> (&mut [T], &mut [T]) {
    let len = slice.len();
    assert!(mid <= len, "mid {mid} out of bounds for slice of len {len}");
    let ptr = slice.as_mut_ptr();

    // SAFETY: mid <= len (asserted), so both halves stay in the allocation and
    // [0, mid) and [mid, len) are disjoint — the two &mut never alias.
    unsafe {
        (
            slice::from_raw_parts_mut(ptr, mid),
            slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}
```

A complete program using the safe API — no `unsafe` in sight for the caller:

```rust title="main.rs — complete, runnable"
use std::slice;

fn split_at_mut<T>(s: &mut [T], mid: usize) -> (&mut [T], &mut [T]) {
    let len = s.len();
    assert!(mid <= len);
    let ptr = s.as_mut_ptr();
    // SAFETY: mid <= len, so the two ranges are disjoint and in-bounds.
    unsafe { (slice::from_raw_parts_mut(ptr, mid), slice::from_raw_parts_mut(ptr.add(mid), len - mid)) }
}

fn main() {
    let mut data = [1, 2, 3, 4, 5];
    let (left, right) = split_at_mut(&mut data, 2);
    left[0] = 100;
    right[0] = 300;
    println!("{data:?}"); // [100, 2, 300, 4, 5]
}
```

Output:

```text
[100, 2, 300, 4, 5]
```

## FFI, direction 1: calling C from Rust

An `extern "C"` block declares functions implemented elsewhere (here, the C
standard library). Calling them is `unsafe` because Rust can't verify C's
contract. Wrap the call in a safe function:

```rust title="src/lib.rs"
use std::os::raw::c_int;

extern "C" {
    fn abs(input: c_int) -> c_int; // from libc
}

pub fn c_abs(x: i32) -> i32 {
    // SAFETY: `abs` is pure over an int with no memory requirements.
    unsafe { abs(x as c_int) as i32 }
}
```

## FFI, direction 2: exposing Rust to C

Mark a function `extern "C"` and `#[no_mangle]` so C code can find and call it by
name. On the Rust side it's an ordinary safe function.

```rust title="src/lib.rs"
use std::os::raw::c_int;

#[no_mangle]
pub extern "C" fn rust_triple(x: c_int) -> c_int {
    x * 3
}
```

A C program could then `extern int rust_triple(int);` and call it after linking
against the compiled Rust library.

## Common mistakes & debugging

### 1. Dereferencing a dangling raw pointer

```rust
fn main() {
    let ptr: *const i32 = {
        let x = 5;
        &x // x is dropped at the end of this block
    };
    // SAFETY: (there is none — this is the bug)
    let value = unsafe { *ptr }; // ✗ undefined behaviour: x is gone
    println!("{value}");
}
```

This may print garbage, crash, or appear to work — classic UB. **The compiler
won't stop you** inside `unsafe`; tools like Miri (`cargo +nightly miri run`) will.
Keep raw pointers derived from data that clearly outlives them.

### 2. A wrong FFI signature

If your `extern "C"` declaration doesn't exactly match the C function's real
signature (wrong argument types, wrong return type), calling it is undefined
behaviour with no error. **Fix:** match the C header precisely, use the
`std::os::raw` / `std::ffi` types (`c_int`, `c_char`, `CStr`, `CString`), and
consider `bindgen` to generate bindings automatically.

### 3. `unsafe` that didn't need to be

Reaching for raw pointers when a slice, iterator, or `Cell` would do. Most
"performance" `unsafe` is unnecessary — the optimizer already elides bounds checks
it can prove are redundant. **Fix:** write safe code first, measure, and only
introduce `unsafe` for a proven, documented reason.

fn read_through_raw(r: &i32) -> i32 {
    let p: *const i32 = r;

    let x = 99;

## Best practices vs anti-patterns

| ✅ Best practice | ❌ Anti-pattern |
| --- | --- |
| Smallest possible `unsafe` block | A whole function marked `unsafe` "to be safe" |
| A `// SAFETY:` comment justifying every block | `unsafe` with no explanation of why it's sound |
| Wrap the unsafe core in a safe, validated API | Exposing raw pointers in your public API |
| Match C signatures exactly; use `std::ffi` types | Guessing at FFI types (silent UB) |
| Verify with Miri and tests | Assuming "it ran once" means it's sound |
| Write safe code first; measure before `unsafe` | Premature `unsafe` for imagined speed |

## Interview questions

<Interview question="Does unsafe turn off the borrow checker?">

No. Inside `unsafe`, the borrow checker, type checker, and lifetime rules all
still apply. `unsafe` only permits five additional operations the compiler cannot
verify (raw-pointer dereference, calling `unsafe`/FFI functions, implementing
`unsafe` traits, mutable statics, and `union` fields). You take on responsibility
for the invariants around those operations — nothing else changes.

</Interview>

<Interview question="What does it mean to build a 'safe abstraction' over unsafe code?">

It means confining `unsafe` to a small, audited core and exposing an API that
callers can use without any `unsafe` and without being able to cause undefined
behaviour, no matter what inputs they pass. `Vec`, `split_at_mut`, and `Mutex` all
do this: unsafe inside, safe outside. The safe wrapper validates inputs (e.g. the
`mid <= len` assert) so the invariants the unsafe core relies on always hold.

</Interview>

<Interview question="Why is calling an FFI function unsafe?">

Because Rust cannot see or verify the foreign function's implementation or
contract. The C side might dereference a pointer you passed, expect a specific
memory layout, mutate through a `const` pointer, or have a signature that doesn't
match your declaration — none of which the Rust compiler can check. You assert, by
writing `unsafe`, that you've verified the contract holds.

</Interview>

<Interview question="What tools help you verify unsafe code?">

`Miri`, an interpreter that detects many kinds of undefined behaviour (out-of-bounds
access, use-after-free, misaligned reads, data races) at run time — invaluable for
testing `unsafe`. Beyond it: `clippy`'s `undocumented_unsafe_blocks` lint, address
and thread sanitizers (`-Z sanitizer`), and `cargo-fuzz` for fuzzing. And of course
ordinary tests exercising the safe wrapper's edges.

</Interview>

## Challenge

Reimplement a small slice of the standard library yourself — `Vec<T>`'s `push` and
`pop` — using `alloc`, raw pointers, and `ptr::write`/`ptr::read`, wrapped in a
safe API. Then run it under `cargo +nightly miri test`. Miri will almost certainly
catch a subtle mistake the first time (a misaligned write, an off-by-one in
capacity, a leaked allocation). Fixing what Miri finds is the single best way to
build real intuition for what "sound `unsafe`" actually requires.
