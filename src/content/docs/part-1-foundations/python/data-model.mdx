The secret to Python's elegance is the **data model**: a set of special methods,
written with double underscores (`__len__`, `__getitem__`, `__repr__`, `__add__`…
— "dunders"), that the interpreter calls on your behalf. You never call
`obj.__len__()`; you write `len(obj)` and Python calls it for you. Implement the
right dunders and your own objects support `len()`, indexing, slicing, iteration,
`in`, arithmetic, and clean printing — using the *same syntax as built-in types*.
This is what Pythonistas mean by "Pythonic". This lesson makes it concrete.

<Aside type="note" title="Prerequisites">
  You should know Python classes (`class`, `__init__`, `self`) and basic
  built-ins (`len`, `for`, `in`). Everything about dunder methods is built up
  here.
</Aside>

## Two methods make a sequence

Implement just `__len__` and `__getitem__`, and your object becomes a full
sequence — `len()`, indexing, slicing, iteration, `in`, and `reversed()` all work,
none of which you wrote.

```python title="main.py — complete, runnable"
from collections import namedtuple

Card = namedtuple("Card", ["rank", "suit"])


class FrenchDeck:
    ranks = [str(n) for n in range(2, 11)] + list("JQKA")
    suits = ["spades", "diamonds", "clubs", "hearts"]

    def __init__(self):
        self._cards = [Card(rank, suit) for suit in self.suits for rank in self.ranks]

    def __len__(self):
        return len(self._cards)

    def __getitem__(self, position):
        return self._cards[position]


deck = FrenchDeck()
print(len(deck))                    # 52
print(deck[0])                      # Card(rank='2', suit='spades')
print(deck[-1])                     # Card(rank='A', suit='hearts')
print(Card("Q", "hearts") in deck)  # True   (in works via iteration)
print(deck[:2])                     # first two cards (slicing works too)
```

Output:

```text
52
Card(rank='2', suit='spades')
Card(rank='A', suit='hearts')
True
[Card(rank='2', suit='spades'), Card(rank='3', suit='spades')]
```

<Aside type="tip" title="You rarely call dunders directly">
  Write `len(x)`, not `x.__len__()`; `x + y`, not `x.__add__(y)`; `repr(x)`, not
  `x.__repr__()`. The interpreter maps the syntax to the dunder. Defining them is
  how you *opt in* to that syntax; calling them directly is a code smell.
</Aside>

## Arithmetic and representation

A numeric type shows off the operator and display dunders. `__repr__` gives an
unambiguous, ideally source-like string; `__add__`/`__mul__` power `+` and `*`;
`__eq__` powers `==`; `__bool__` decides truthiness.

```python title="main.py — complete, runnable"
import math


class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __repr__(self):
        return f"Vector({self.x!r}, {self.y!r})"

    def __eq__(self, other):
        if not isinstance(other, Vector):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)

    def __abs__(self):
        return math.hypot(self.x, self.y)

    def __bool__(self):
        return bool(abs(self))

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)


print(Vector(3, 4))              # Vector(3, 4)      (via __repr__)
print(abs(Vector(3, 4)))         # 5.0
print(Vector(1, 2) + Vector(2, 4))  # Vector(3, 6)
print(Vector(2, 3) * 3)          # Vector(6, 9)
print(bool(Vector(0, 0)))        # False
print(Vector(1, 2) == Vector(1, 2))  # True
```

Output:

```text
Vector(3, 4)
5.0
Vector(3, 6)
Vector(6, 9)
False
True
```

## `__repr__` vs `__str__`

`__repr__` is for developers — unambiguous, shown in the REPL and by `repr()`.
`__str__` is for end users — readable, used by `print()` and `str()`. If you only
define one, make it `__repr__` (it's the fallback for `__str__`).

```python title="main.py — complete, runnable"
class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    def __repr__(self):
        return f"Temperature(celsius={self.celsius!r})"

    def __str__(self):
        return f"{self.celsius}°C"


t = Temperature(21)
print(str(t))   # 21°C            (human-friendly)
print(repr(t))  # Temperature(celsius=21)   (unambiguous)
print(t)        # 21°C            (print uses __str__)
```

Output:

```text
21°C
Temperature(celsius=21)
21°C
```

## Common mistakes & debugging

### 1. Defining `__eq__` but forgetting `__hash__`

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)

# Point() is now UNHASHABLE — can't be a dict key or set member:
# {Point(1, 2)}  ->  TypeError: unhashable type: 'Point'
```

Defining `__eq__` sets `__hash__` to `None`. **Fix:** if instances are immutable
and should be hashable, also define `__hash__` (e.g. `return hash((self.x,
self.y))`), or use `@dataclass(frozen=True)` which does both for you.

### 2. Returning `False` instead of `NotImplemented`

In `__eq__`, when the other operand is an unexpected type, return
`NotImplemented` (not `False`). This lets Python try the reflected operation and
keeps comparisons symmetric. Returning `False` can make `a == b` and `b == a`
disagree.

### 3. A misleading `__repr__`

`__repr__` should be unambiguous and, where practical, look like the code that
recreates the object (`Vector(3, 4)`). A vague `<Vector object>` wastes the best
debugging tool you have. Use `!r` in the f-string to repr the fields.

<Interview question="Exercise 3 — Make an object iterable via __getitem__.">

Write a `Playlist` that wraps a list of songs and supports iteration and `len`.
Complete program.

**Solution — complete, runnable:**

```python
class Playlist:
    def __init__(self, songs):
        self._songs = list(songs)

    def __len__(self):
        return len(self._songs)

    def __getitem__(self, i):
        return self._songs[i]


pl = Playlist(["a", "b", "c"])
print(len(pl))            # 3
print(pl[1])             # b
print("c" in pl)         # True
print(list(pl))          # ['a', 'b', 'c']
```

Iteration, `in`, and slicing all come from the single `__getitem__`.

</Interview>

## Best practices vs anti-patterns

| ✅ Best practice | ❌ Anti-pattern |
| --- | --- |
| Write `len(x)` / `x + y`; let dispatch happen | Calling `x.__len__()` / `x.__add__(y)` directly |
| Unambiguous `__repr__` (looks like source) | Leaving the default `<Foo object at 0x...>` |
| Return `NotImplemented` for unknown operand types | Returning `False`/raising in `__eq__` |
| Define `__hash__` when you define `__eq__` (if hashable) | `__eq__` alone, silently breaking hashing |
| `@dataclass` for plain data holders | Hand-writing `__init__`/`__repr__`/`__eq__` boilerplate |
| `__getitem__`/`__len__` for sequence-like types | Custom `.get_item()` / `.size()` methods |

## Interview questions

<Interview question="What are dunder methods and why does Python use them?">

Dunder ("double underscore") methods like `__len__`, `__getitem__`, `__add__` and
`__repr__` are the hooks of Python's data model. The interpreter calls them in
response to syntax and built-ins: `len(x)` calls `x.__len__()`, `a + b` calls
`a.__add__(b)`, `x[i]` calls `x.__getitem__(i)`. They let user-defined types
integrate seamlessly with the language's operators and functions, so your objects
behave like built-ins. You implement them to opt in; you rarely call them directly.

</Interview>

<Interview question="What is the difference between __repr__ and __str__?">

`__repr__` is meant for developers: an unambiguous representation, ideally one that
looks like valid Python to recreate the object; it's what the REPL and `repr()`
show. `__str__` is meant for end users: a readable, possibly less precise string,
used by `print()` and `str()`. If `__str__` is absent, Python falls back to
`__repr__`, so defining `__repr__` is the higher priority.

</Interview>

<Interview question="Why can defining __eq__ break using your objects in sets or dicts?">

When you define `__eq__`, Python sets the class's `__hash__` to `None`, making
instances unhashable — so they can't be dict keys or set members (you'd get
`TypeError: unhashable type`). This is deliberate: two objects that are equal must
have the same hash, and Python can't guarantee that automatically. If your objects
are immutable and should be hashable, define `__hash__` too, or use a frozen
dataclass.

</Interview>

<Interview question="What does returning NotImplemented from __eq__ or __add__ do?">

It tells Python "I don't know how to handle this operand type", prompting the
interpreter to try the reflected method on the other operand (e.g. `other.__radd__`
or `other.__eq__`), and if that also declines, to fall back to a default (identity
for `==`, or `TypeError` for arithmetic). This keeps operations symmetric and
extensible. Note `NotImplemented` (a special singleton) is different from
`NotImplementedError` (an exception).

</Interview>
