Your language can [make decisions](/systems-programming-academy/part-7-interpreter/conditionals/),
but its conditions are thin: a single comparison. Real conditions combine them —
"in range" is `x >= lo and x <= hi`; a leap year is a small tangle of `and`, `or`,
and `not`. This lesson adds those three operators, and in doing so reuses the most
important idea from the last lesson: **short-circuit evaluation** — the same
laziness that makes `if` work.

<Aside type="note" title="and and or are lazy, just like if">
  `and` and `or` are not ordinary operators. `a and b` evaluates `a`, and if `a`
  is already false, it returns false **without evaluating `b`** — because the
  answer is settled. `a or b` skips `b` when `a` is already true. This
  short-circuiting is exactly the branch-skipping of `if`, and it's a *correctness*
  property: `x != 0 and 100 / x > 1` is safe precisely because `100 / x` is never
  evaluated when `x` is 0. Booleans stay integers (`1`/`0`, any nonzero truthy),
  so results stay exact and byte-identical.
</Aside>

## No new tokens — just keywords

`and`, `or`, and `not` need nothing from the lexer: they tokenize as ordinary
identifiers, and the parser recognizes them by name (just like `if`/`then`/`else`).
What they *do* need is three new precedence levels, slotted below comparison:

```text
expr    := 'if' ... | or_expr
or_expr  := and_expr ('or' and_expr)*      ← loosest
and_expr := not_expr ('and' not_expr)*
not_expr := 'not' not_expr | comparison    ← 'not' binds tighter than and/or
comparison := add (cmpop add)?
```

This ordering — `or` looser than `and` looser than `not` looser than comparison —
is the universal convention, and it's why `x >= lo and x <= hi` groups the two
comparisons *before* the `and`, and why `1 or 0 and 0` means `1 or (0 and 0)`.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="boollang.py"
def or_expr(self) -> Expr:               # and_expr ('or' and_expr)*
    left = self.and_expr()
    while self._is_kw("or"):
        self.pos += 1
        left = Expr("O", left=left, right=self.and_expr())
    return left

def and_expr(self) -> Expr:              # not_expr ('and' not_expr)*
    left = self.not_expr()
    while self._is_kw("and"):
        self.pos += 1
        left = Expr("A", left=left, right=self.not_expr())
    return left

def not_expr(self) -> Expr:              # 'not' not_expr | comparison
    if self._is_kw("not"):
        self.pos += 1
        return Expr("N", left=self.not_expr())
    return self.comparison()
```
</TabItem>
<TabItem label="Go">
```go title="bool.go"
func (p *parser) orExpr() (*Expr, error) {     // andExpr ('or' andExpr)*
	left, _ := p.andExpr()
	for p.isKw("or") {
		p.pos++
		right, err := p.andExpr()
		if err != nil { return nil, err }
		left = &Expr{Kind: 'O', L: left, R: right}
	}
	return left, nil
}

func (p *parser) notExpr() (*Expr, error) {    // 'not' notExpr | comparison
	if p.isKw("not") {
		p.pos++
		inner, err := p.notExpr()
		if err != nil { return nil, err }
		return &Expr{Kind: 'N', L: inner}, nil
	}
	return p.comparison()
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
fn or_expr(&mut self) -> Result<Expr, String> {   // and_expr ('or' and_expr)*
    let mut left = self.and_expr()?;
    while self.is_kw("or") {
        self.pos += 1;
        left = Expr::Or(Box::new(left), Box::new(self.and_expr()?));
    }
    Ok(left)
}

fn not_expr(&mut self) -> Result<Expr, String> {  // 'not' not_expr | comparison
    if self.is_kw("not") {
        self.pos += 1;
        return Ok(Expr::Not(Box::new(self.not_expr()?)));
    }
    self.comparison()
}
```
</TabItem>
</Tabs>

Note `not_expr` recurses on itself, so `not not 3` parses cleanly as `not (not 3)`.

## Evaluation: short-circuit, by hand

The eval cases are where the laziness lives. Unlike arithmetic (which evaluates
both operands up front), `and` and `or` evaluate the left, decide whether the
answer is already known, and **only then** touch the right:

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="boollang.py"
if e.kind == "N":                                  # not
    return 1 if _as_int(evaluate(e.left, env)) == 0 else 0
if e.kind in ("A", "O"):
    left = _as_int(evaluate(e.left, env))
    if e.kind == "A" and left == 0:
        return 0                                   # short-circuit: skip the right
    if e.kind == "O" and left != 0:
        return 1                                   # short-circuit: skip the right
    return 1 if _as_int(evaluate(e.right, env)) != 0 else 0
```
</TabItem>
<TabItem label="Go">
```go title="bool.go"
if e.Kind == 'N' { // not: negate truthiness
	v, err := Eval(e.L, env)
	if err != nil { return Value{}, err }
	n, _ := asInt(v)
	return Value{Kind: 'i', Int: 1 - truthy(n)}, nil
}
lv, err := Eval(e.L, env)
if err != nil { return Value{}, err }
left, _ := asInt(lv)
if e.Kind == 'A' && left == 0 {
	return Value{Kind: 'i', Int: 0}, nil  // short-circuit: skip the right
}
if e.Kind == 'O' && left != 0 {
	return Value{Kind: 'i', Int: 1}, nil  // short-circuit: skip the right
}
rv, _ := Eval(e.R, env)
right, _ := asInt(rv)
return Value{Kind: 'i', Int: truthy(right)}, nil
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
Expr::And(l, r) => {
    if as_int(eval(l, env)?)? == 0 {
        Ok(Value::Int(0))                          // short-circuit: skip the right
    } else {
        Ok(Value::Int(if as_int(eval(r, env)?)? != 0 { 1 } else { 0 }))
    }
}
Expr::Or(l, r) => {
    if as_int(eval(l, env)?)? != 0 {
        Ok(Value::Int(1))                          // short-circuit: skip the right
    } else {
        Ok(Value::Int(if as_int(eval(r, env)?)? != 0 { 1 } else { 0 }))
    }
}
Expr::Not(inner) => Ok(Value::Int(if as_int(eval(inner, env)?)? == 0 { 1 } else { 0 })),
```
</TabItem>
</Tabs>

<Aside type="tip" title="Why you can't write and/or as a function">
  In this language, a *function* evaluates all its arguments before the body runs
  (that's how the call machinery from the functions lesson works). So `and`/`or`
  **cannot** be ordinary functions — a function `myAnd(a, b)` would evaluate `b`
  before it ever ran, defeating the short-circuit. They must be built into the
  evaluator as *special forms* that control evaluation. This is a deep reason
  languages have keywords like `and`/`if` distinct from functions: some constructs
  need to decide *whether* to evaluate their operands, and only the evaluator can
  do that.
</Aside>

## A real predicate: leap year

With `and`, `or`, and `not`, the conditions become genuinely expressive. Here is
the full Gregorian leap-year rule — divisible by 4, but not by 100 unless also by
400 — written in *your* language (using `y - y/4*4 == 0` as "divisible by 4", since
we have no modulo yet):

```text
leap(y) = if y - y/4*4 == 0 and (y - y/100*100 != 0 or y - y/400*400 == 0) then 1 else 0
leap(2000)   => 1     # divisible by 400
leap(1900)   => 0     # divisible by 100 but not 400
leap(2024)   => 1     # divisible by 4, not by 100
```

All three implementations produce this byte-for-byte. A non-trivial real-world
predicate, running on the interpreter you grew from a character scanner — the
boolean operators are what make conditions like this readable.

    toks, i = [], 0
    while i < len(s):

        if c in " \\t": i += 1
        elif c == "=" and nx == "=": toks.append("=="); i += 2
        elif c in "!<>" and nx == "=": toks.append(c + "="); i += 2
        elif c in "<>": toks.append(c); i += 1
        elif c in "=,+-*/()": toks.append(c); i += 1
        elif c.isdigit():

class P:
    def __init__(s, t): s.t, s.i = t, 0
    def peek(s): return s.t[s.i] if s.i < len(s.t) else None
    def nxt(s):
        v = s.peek(); s.i += 1; return v
    def iskw(s, k): return s.peek() == ("id", k)
    def expr(s):
        if s.iskw("if"):
            s.nxt(); cond = s.expr()
            if not s.iskw("then"): raise ValueError("expected 'then'")
            s.nxt(); then = s.expr()
            if not s.iskw("else"): raise ValueError("expected 'else'")

    def factor(s):
        t = s.nxt()
        if isinstance(t, int): return t
        if isinstance(t, tuple):
            if s.peek() == "(":
                s.nxt(); args = []

def ev(n, env):
    if isinstance(n, int): return n
    k = n[0]
    if k == "var":
        v = env.get(n[1])
        if v is None: raise ValueError(f"undefined variable {n[1]!r}")

    if k == "neg": return -asint(ev(n[1], env))
    if k == "not": return 1 if asint(ev(n[1], env)) == 0 else 0
    if k == "and":

        return 1 if asint(ev(n[2], env)) != 0 else 0
    if k == "or":

        return 1 if asint(ev(n[2], env)) != 0 else 0

## Exercises

**1. Add `xor` (exclusive or) — and notice it can't short-circuit (all three).** `a
xor b` is true when exactly one side is true. Implement it. Why, unlike `and` and
`or`, must `xor` *always* evaluate both operands?

<details>
<summary>Solution sketch</summary>

`xor` needs both values no matter what: knowing only the left tells you nothing
about the result, since `left xor b` flips with `b` either way. So it has no
short-circuit form — it evaluates both sides and returns `1` if they differ. That
contrast is illuminating: `and`/`or` can short-circuit *because* one operand can
decide the whole answer, and `xor` can't *because* neither can. Short-circuiting
isn't a property of "boolean operators" in general — it's specific to operators
with a dominating value.

</details>

**2. Prove short-circuit ordering matters (research + one language).** Write
`10 / x > 0 and x != 0` — the guard in the *wrong* order — and call it with `x = 0`.
What happens, and what does it teach about how programmers rely on short-circuit
evaluation in real code?

<details>
<summary>Solution sketch</summary>

It divides by zero and errors, because `and` evaluates left-to-right: it reaches
`10 / x` before the guarding `x != 0`. The safe form puts the guard first
(`x != 0 and 10 / x > 0`), so the dangerous operation is only reached when it's
safe. Real code depends on this constantly — `if p != null and p.value > 0`,
`if i < len and arr[i] == x` — the guard *must* come first, and short-circuit
evaluation is the guarantee that makes it work. Order matters precisely because
evaluation is lazy and left-to-right.

</details>

**3. Represent booleans as their own type (any language).** Right now `true` and
`false` are just `1` and `0`. Add real boolean values distinct from integers, so
`1 and 2` becomes a type error rather than truthy arithmetic. What do you gain, and
what does it cost the byte-identical simplicity?

<details>
<summary>Solution sketch</summary>

You'd add a `Bool` variant to the value type alongside `Int`, make comparisons and
`and`/`or`/`not` produce `Bool`, and make `if` require a `Bool` condition — turning
`if 5 then ...` into a type error instead of silently truthy. The gain is safety:
the language catches nonsense like `1 and 2` or `if <a function> then`. The cost is
complexity (a richer value type, more error cases) and that the neat "booleans are
just integers" trick — which kept every result exact and byte-identical here — goes
away. It's the classic trade every language makes between simplicity and a stricter,
safer type system.

</details>

## Interview corner

<Interview question="What is short-circuit evaluation, and why does it matter beyond performance?">

Short-circuit evaluation means a boolean operator stops as soon as its result is
determined: `a and b` returns false without evaluating `b` if `a` is false, and
`a or b` returns true without evaluating `b` if `a` is true. It matters far beyond
saving a little work — it's a *correctness* mechanism programmers rely on
constantly. The classic pattern is a guard: `p != null and p.field > 0`, or
`i < len and arr[i] == x`. The right side is only *safe* to evaluate when the left
is true, and short-circuiting guarantees it's only reached then — so the null
dereference or out-of-bounds access never happens. That's why order matters:
`arr[i] == x and i < len` would crash, because `and` evaluates left to right and
reaches the indexing before the bounds check. Implementing it teaches a deeper
lesson too: short-circuit operators can't be ordinary functions, because a function
evaluates all its arguments before running. They have to be *special forms* the
evaluator handles directly, deciding whether to evaluate the second operand at all —
the same category as `if`. So a language distinguishes keywords like `and`, `or`,
and `if` from functions precisely because some constructs must control what gets
evaluated, and that control is exactly what makes guards, laziness, and safe
conditionals possible.

</Interview>
