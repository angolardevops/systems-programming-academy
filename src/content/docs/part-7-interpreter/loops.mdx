So far every program was one expression, and the interpreter *echoed* its value. A
real language is different: it **does things** — it produces output and it repeats
work. This lesson makes that leap. A program becomes a sequence of **statements**
separated by `;`, grouped with `{ }` **blocks**; a `print` statement produces
output; and a `while` loop repeats. Suddenly you can write actual programs:

```text
i = 1;
while i <= 5 do { print i; i = i + 1 }
```

which prints `1 2 3 4 5`. The language you grew from a character scanner now runs
recognizable code.

<Aside type="note" title="Loops add convenience, not power">
  Here is a subtle but important point: `while` adds **no computational power**.
  The language became Turing-complete back at
  [conditionals](/systems-programming-academy/part-7-interpreter/conditionals/) —
  recursion can already express any repetition. What a loop adds is *convenience*
  (an explicit, readable iteration) and *efficiency* (it runs in constant stack
  space, where deep recursion would grow the stack). Many language features are
  like this: comfort and reach layered on a small, already-universal core. What's
  genuinely new in this lesson is **statements and output** — the shift from
  computing a value to *running a program*.
</Aside>

## A program is now a list of statements

The biggest change is the model. Before, `run_program` returned `"line => value"`
for each line. Now it tokenizes the *whole* source (newlines are just whitespace),
parses a **sequence of statements** separated by `;`, executes them against one
environment, and returns only what `print` emitted. Statements are a new layer
above expressions:

```text
program := stmt (';' stmt)*
stmt := 'print' expr
      | 'while' expr 'do' stmt
      | '{' stmt (';' stmt)* '}'     (a block — a statement made of statements)
      | name '(' params ')' '=' expr  (function definition)
      | name '=' expr                 (assignment)
      | expr                          (expression statement; value discarded)
```

The elegant part: a **block is itself a statement** — `{ s1; s2 }` — so `while`'s
body is "one statement" that happens to be a block of many. That recursion (a
statement can contain statements) is what gives you nesting for free: loops inside
loops, blocks inside blocks.

## `print` and `while`: doing and repeating

Execution now takes an *output accumulator*. `print` evaluates its expression and
appends the result; `while` evaluates its condition and, while truthy, runs its
body — over and over, mutating the environment each time:

def execute(stmt: Stmt, env: Scope, out: list[str]) -> None:
    if stmt.kind == "p":                                  # print

    elif stmt.kind == "w":                                # while

    elif stmt.kind == "b":                                # block

            execute(s, env, out)
    elif stmt.kind == "a":                                # assignment (mutation)

`while` works because assignment *mutates* the shared environment: the loop's body
changes the counter (`i = i + 1`), and the next condition check sees the new value.
That is the imperative heart — state that changes over time, driven by iteration —
and it rests entirely on the environment you built back in the variables lesson.

def tokenize(s):
    toks, i = [], 0
    while i < len(s):
        c = s[i]; nx = s[i + 1] if i + 1 < len(s) else ""
        if c in " \\t\\n": i += 1
        elif c == "=" and nx == "=": toks.append("=="); i += 2
        elif c in "!<>" and nx == "=": toks.append(c + "="); i += 2
        elif c in "<>": toks.append(c); i += 1
        elif c in "=;{},+-*/()": toks.append(c); i += 1
        elif c.isdigit():
            j = i
            while j < len(s) and s[j].isdigit(): j += 1
            toks.append(int(s[i:j])); i = j
        elif c.isalpha() or c == "_":

            toks.append(("id", s[i:j])); i = j
        else: raise ValueError(f"unexpected character {c!r}")
    return toks

class P:
    def __init__(s, t): s.t, s.i = t, 0
    def peek(s): return s.t[s.i] if s.i < len(s.t) else None
    def at(s, k): return s.t[s.i + k] if s.i + k < len(s.t) else None
    def nxt(s):
        v = s.peek(); s.i += 1; return v
    def iskw(s, k): return s.peek() == ("id", k)
    def program(s):
        stmts = []
        while True:
            while s.peek() == ";": s.i += 1
            if s.peek() is None: return stmts
            stmts.append(s.stmt())
            if s.peek() is not None and s.peek() != ";":
                raise ValueError("expected ';' between statements")
    def stmt(s):
        if s.iskw("print"): s.nxt(); return ("print", s.expr())
        if s.iskw("while"):
            s.nxt(); cond = s.expr()
            if not s.iskw("do"): raise ValueError("expected 'do'")
            s.nxt(); return ("while", cond, s.stmt())
        if s.peek() == "{":
            s.nxt(); body = []
            while True:
                while s.peek() == ";": s.i += 1
                if s.peek() == "}": s.nxt(); break
                if s.peek() is None: raise ValueError("unclosed '{'")
                body.append(s.stmt())
                if s.peek() is not None and s.peek() not in (";", "}"):
                    raise ValueError("expected ';' or '}'")
            return ("block", body)
        if isinstance(s.peek(), tuple) and s.at(1) == "(" and s.is_fndef():
            name = s.nxt()[1]; s.nxt()
            params = []
            if s.peek() != ")":
                while True:
                    params.append(s.nxt()[1])
                    sep = s.nxt()
                    if sep == ",": continue
                    if sep == ")": break
                    raise ValueError("bad params")
            else: s.nxt()
            if s.nxt() != "=": raise ValueError("expected '='")
            return ("fndef", name, params, s.expr())
        if isinstance(s.peek(), tuple) and s.at(1) == "=":
            name = s.nxt()[1]; s.nxt()
            return ("assign", name, s.expr())
        return ("expr", s.expr())
    def is_fndef(s):
        depth, k = 0, 1
        while s.at(k) is not None:
            if s.at(k) == "(": depth += 1
            elif s.at(k) == ")":
                depth -= 1
                if depth == 0: return s.at(k + 1) == "="
            k += 1
        return False
    def expr(s):
        if s.iskw("if"):
            s.nxt(); cond = s.expr()
            if not s.iskw("then"): raise ValueError("expected 'then'")
            s.nxt(); then = s.expr()
            if not s.iskw("else"): raise ValueError("expected 'else'")
            s.nxt(); return ("if", cond, then, s.expr())
        return s.orx()
    def orx(s):
        n = s.andx()
        while s.iskw("or"): s.nxt(); n = ("or", n, s.andx())
        return n
    def andx(s):
        n = s.notx()
        while s.iskw("and"): s.nxt(); n = ("and", n, s.notx())
        return n
    def notx(s):
        if s.iskw("not"): s.nxt(); return ("not", s.notx())
        return s.cmp()
    def cmp(s):
        left = s.add()
        if s.peek() in CMP:
            op = s.nxt(); return ("cmp", op, left, s.add())
        return left
    def add(s):
        n = s.term()
        while s.peek() in ("+", "-"):
            op = s.nxt(); n = (op, n, s.term())
        return n
    def term(s):
        n = s.factor()
        while s.peek() in ("*", "/"):
            op = s.nxt(); n = (op, n, s.factor())
        return n
    def factor(s):
        t = s.nxt()
        if isinstance(t, int): return t
        if isinstance(t, tuple):
            if s.peek() == "(":
                s.nxt(); args = []
                if s.peek() != ")":
                    args.append(s.expr())
                    while s.peek() == ",": s.nxt(); args.append(s.expr())
                if s.nxt() != ")": raise ValueError("expected ')'")

            if s.nxt() != ")": raise ValueError("expected ')'")

class Scope:
    def __init__(s, parent=None): s.v, s.parent = {}, parent
    def get(s, n):
        sc = s
        while sc:
            if n in sc.v: return sc.v[n]
            sc = sc.parent
        return None

def asint(v):
    if isinstance(v, int): return v
    raise ValueError("cannot do arithmetic on a function")

def ev(n, env):
    if isinstance(n, int): return n
    k = n[0]
    if k == "var":
        v = env.get(n[1])
        if v is None: raise ValueError(f"undefined variable {n[1]!r}")
        return v
    if k == "neg": return -asint(ev(n[1], env))
    if k == "not": return 1 if asint(ev(n[1], env)) == 0 else 0
    if k == "and":
        if asint(ev(n[1], env)) == 0: return 0
        return 1 if asint(ev(n[2], env)) != 0 else 0
    if k == "or":
        if asint(ev(n[1], env)) != 0: return 1
        return 1 if asint(ev(n[2], env)) != 0 else 0
    if k == "cmp":
        a, b = asint(ev(n[2], env)), asint(ev(n[3], env))
        return 1 if CMPF[n[1]](a, b) else 0
    if k == "if":
        return ev(n[2], env) if asint(ev(n[1], env)) != 0 else ev(n[3], env)
    if k == "call":
        fn = env.get(n[1])
        if not (isinstance(fn, tuple) and fn[0] == "fn"): raise ValueError(f"{n[1]!r} is not a function")
        _, params, body, defenv = fn
        args = [ev(a, env) for a in n[2]]
        if len(args) != len(params): raise ValueError(f"{n[1]!r} arity mismatch")
        call = Scope(defenv)
        for p, a in zip(params, args): call.v[p] = a
        return ev(body, call)
    a, b = asint(ev(n[1], env)), asint(ev(n[2], env))
    if k == "+": return a + b
    if k == "-": return a - b
    if k == "*": return a * b
    if b == 0: raise ValueError("division by zero")
    q = abs(a) // abs(b)
    return -q if (a < 0) != (b < 0) else q

def execute(stmt, env, out):
    k = stmt[0]
    if k == "fndef": env.v[stmt[1]] = ("fn", stmt[2], stmt[3], env)
    elif k == "assign": env.v[stmt[1]] = ev(stmt[2], env)
    elif k == "print": out.append(str(ev(stmt[1], env)))
    elif k == "while":
        while asint(ev(stmt[1], env)) != 0: execute(stmt[2], env, out)
    elif k == "block":
        for st in stmt[1]: execute(st, env, out)
    else: ev(stmt[1], env)

def run(program):
    env, out = Scope(), []
    try:
        for stmt in P(tokenize(program)).program():
            execute(stmt, env, out)
    except ValueError as e:
        out.append("error: " + str(e))
    for line in out: print(line)

## Exercises

**1. Add an `if`/`else` *statement* (all three).** There's already an `if`
*expression* (`if c then a else b`), but a statement form — `if c do { ... }` or
`if c do { ... } else { ... }` — lets a branch *run statements* (prints, loops)
rather than produce a value. How does it differ from the expression `if`, and why
do many languages have both?

<details>
<summary>Solution sketch</summary>

The expression `if` *returns a value* and both branches must produce one; the
statement `if` *performs actions* and its branches are statements that may print,
loop, or assign — with an optional `else`. You add a new statement kind that
evaluates the condition and executes the taken block (or the else block, or
nothing). Languages often have both because the two jobs differ: choosing a value
(`x = if cond then 1 else 2`) versus choosing what to *do* (`if ready do { launch }`).
Some languages unify them by making blocks yield values; others keep them separate.

</details>

**2. Add a `for` loop that desugars to `while` (research + one language).** Give the
language `for i = 1 to 10 do { ... }`. The cleanest implementation doesn't add a new
execution case at all — it builds the same tree a `while` would. Show the
desugaring, and explain why it needs no changes to the evaluator.

<details>
<summary>Solution sketch</summary>

Parse `for i = lo to hi do body` and construct the statements `i = lo; while i <= hi
do { body; i = i + 1 }` — the exact tree the equivalent `while` produces. Because the
`for` never becomes a distinct runtime concept (only a parsing convenience that
expands into assignment + `while`), the evaluator needs no new case. This is
*desugaring* again, the same technique as compound assignment earlier: new syntax as
a transformation into existing syntax, which keeps the evaluator small as the surface
language grows.

</details>

**3. Add a `break` statement (any language).** Let a `break` inside a `while` stop
the loop early. Why is this harder than `print` or assignment — what does the
evaluator need that it didn't before?

<details>
<summary>Solution sketch</summary>

`break` needs a way to *interrupt* the normal top-to-bottom flow of statement
execution: when a `break` runs, control must jump out of the enclosing loop, skipping
the rest of the body. Unlike `print`, which just does its thing and returns, `break`
changes *what executes next*. Interpreters implement this with a control-flow signal
— a special return value, an exception, or a status the loop checks after each
statement — that unwinds out to the loop and stops it. It's the first construct that
isn't "evaluate and continue", and it previews how `return`, `continue`, and
exceptions all work: non-local control flow the evaluator must thread through.

</details>

## Interview corner

<Interview question="What changes in an interpreter when you go from an expression evaluator to an imperative language with statements, output, and loops?">

Three related shifts. First, a new syntactic layer: **statements** sit above
expressions. An expression *produces a value*; a statement *performs an action* —
print, assign, loop — and a program is a sequence of them, usually separated by `;`
and grouped into blocks with `{ }`. The parser gains a statement grammar on top of
the expression grammar, and crucially a block is itself a statement, which gives
nesting (loops in loops) for free. Second, **effects**: the evaluator's job is no
longer just to compute and return a value but to *do things in order*, so execution
threads an output channel (for `print`) and relies on the environment being mutable
(so assignment changes state that later statements and loop iterations observe).
Third, **iteration**: a `while` re-evaluates its condition and re-runs its body
against that mutating environment, which is what makes a loop progress and
terminate. Notably, loops add no computational power — recursion plus conditionals
were already Turing-complete — so this is about *ergonomics and I/O*, not
capability. The deep structure is unchanged: it's still lex, parse, and walk a tree;
you've just added a statement layer and let the walk have effects.

</Interview>

## The track ahead

The language is now genuinely usable: variables, functions, closures, conditionals,
boolean logic, and now statements, output, and loops — enough to write FizzBuzz,
compute sequences, and print results, built and tested three ways, byte-identical by
its output. What remains is reach: strings and data structures for richer values,
`break`/`return` for non-local control flow, and better error messages with line
numbers. But you have now built, from a character scanner, a small imperative
programming language that runs real programs — the mystery of "how does a language
work" fully dissolved into code you wrote yourself.
