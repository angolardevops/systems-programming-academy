The [expression evaluator](/systems-programming-academy/part-7-interpreter/expressions/)
computes `2 * (3 + 4)` and forgets it instantly. A *language* remembers: you name
a value and use it later. This lesson adds that — variables, assignment, and the
one structure that makes them work, an **environment** — and in doing so turns a
calculator into a small programming language:

```text
x = 5
y = x * 2 + 1     ← reads x, binds y
y - x             ← 6
```

<Aside type="note" title="The environment is the whole idea">
  Everything here rests on one concept: an **environment**, a `name → value` map
  threaded through evaluation. Assignment writes to it; a variable reference reads
  from it. That's it — but this humble map is the seed of *everything* that
  follows in the track. Add functions and each call gets its own environment
  chained to an enclosing one; that chain is exactly how **scopes**, **closures**,
  and the **call stack** work. Build the environment now and you've built the
  skeleton of a real interpreter's memory.
</Aside>

## Three small additions, one per stage

A variable touches each of the three stages exactly once — the clean sign of a
well-separated design. The **lexer** learns to read identifiers and `=`; the
**parser** learns that a name can be an expression (a *variable reference*) and
that `name = expr` is an *assignment statement*; the **evaluator** gains an
environment to read and write.

Reading an identifier is the same "group characters that belong together" loop as
reading a multi-digit number — start on a letter, keep going while the character
is a letter, digit, or underscore.

### The parser: a variable is an expression; `=` is a statement

A variable reference slots into `factor` right beside a number — both are atoms.
And a whole new node type appears: the **statement**. A program is a sequence of
statements, each either an assignment or a bare expression.

```rust title="src/lib.rs"
pub fn parse_stmt(tokens: Vec<Token>) -> Result<Stmt, String> {
    // `name = expr` is an assignment; anything else is a bare expression.
    let is_assign = matches!(tokens.first(), Some(Token::Ident(_)))
        && matches!(tokens.get(1), Some(Token::Assign));
    let mut p = Parser { tokens, pos: 0 };
    let stmt = if is_assign {
        let name = match p.next() { Some(Token::Ident(n)) => n, _ => unreachable!() };
        p.pos += 1;                        // consume '='
        Stmt::Assign(name, p.expr()?)
    } else {
        Stmt::Expr(p.expr()?)
    };

}
```

Distinguishing assignment from expression with a two-token look-ahead (`Ident`
then `=`) is the simplest possible statement parser. Bigger languages generalize
this, but the shape is the same: decide what *kind* of statement this is, then
parse its parts.

### The evaluator: an environment to read and write

Evaluation now carries an environment. A variable reference looks its name up
(erroring if unbound); an assignment evaluates the right-hand side, stores it, and
returns it. And because statements share one environment, **state persists**.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="vars.py"
def evaluate(e: Expr, env: Env) -> int:
    ...
    if e.kind == "v":                                  # variable reference
        if e.name not in env:
            raise CalcError(f"undefined variable '{e.name}'")
        return env[e.name]
    ...

def execute(stmt: Stmt, env: Env) -> int:
    if stmt.kind == "a":                               # assignment
        v = evaluate(stmt.expr, env)
        env[stmt.name] = v                             # write to the environment
        return v
    return evaluate(stmt.expr, env)
```

```rust title="src/lib.rs"
pub fn eval(e: &Expr, env: &Env) -> Result<i64, String> {
    match e {
        // variable reference
        Expr::Var(name) => env.get(name).copied()
            .ok_or_else(|| format!("undefined variable '{name}'")),
        ...
    }
}

pub fn exec(stmt: &Stmt, env: &mut Env) -> Result<i64, String> {
    match stmt {
        Stmt::Assign(name, e) => {          // assignment
            let v = eval(e, env)?;
            env.insert(name.clone(), v);    // write to the environment
            Ok(v)
        }
        Stmt::Expr(e) => eval(e, env),
    }
}
```
</TabItem>

Notice the ownership shapes: Rust makes the mutation explicit — `eval` takes
`&Env` (read-only), `exec` takes `&mut Env` (it writes), and the compiler enforces
that split. Go and Python pass the map by reference and mutate it freely. Same
behaviour; Rust just makes "who can write" visible in the type.

## State in action

Run a whole program — one statement per line, sharing one environment — and the
memory shows:

```text
$ vars "x = 5; y = x * 2 + 1; y = y + 100; y - x"
x = 5  =>  5
y = x * 2 + 1  =>  11
y = y + 100  =>  111
y - x  =>  106
```

The third line is the quiet lesson: `y = y + 100` reads the *old* `y` (11), adds
100, and stores the result back (111). Assignment is not equation-solving — it's
"evaluate the right side in the current environment, then rebind the name." That
right-then-left order is why `x = x + 1` makes sense at all, and it's identical in
every language here.

class P:
    def __init__(self, t): self.t, self.i = t, 0
    def peek(self): return self.t[self.i] if self.i < len(self.t) else None
    def nxt(self):
        v = self.peek(); self.i += 1; return v
    def expr(self):
        n = self.term()
        while self.peek() in ("+", "-"):
            op = self.nxt(); n = (op, n, self.term())
        return n
    def term(self):
        n = self.factor()
        while self.peek() in ("*", "/"):
            op = self.nxt(); n = (op, n, self.factor())
        return n

        t = self.nxt()
        if isinstance(t, int): return t
        if isinstance(t, tuple): return t          # ('var', name)
        if t == "-": return ("neg", self.factor())
        if t == "(":
            n = self.expr()
            if self.nxt() != ")": raise ValueError("expected ')'")
            return n
        raise ValueError("unexpected end of input" if t is None else f"unexpected {t!r}")

    if isinstance(n, int): return n
    if n[0] == "var":
        if n[1] not in env: raise ValueError(f"undefined variable {n[1]!r}")

    if n[0] == "neg": return -ev(n[1], env)

    if n[0] == "-": return a - b
    if n[0] == "*": return a * b

## Exercises

**1. Add compound assignment `+=` (all three).** Let `x += 3` mean `x = x + 3`.
Where is the cleanest place to implement it — the lexer, the parser, or by
*desugaring* it in the parser into the assignment you already have?

<details>
<summary>Solution sketch</summary>

The elegant move is *desugaring*: parse `x += 3` and build the exact same tree as
`x = x + 3` — an assignment whose right side is `x + 3`. Then the evaluator needs
no changes at all, because `+=` was never a real concept to it, only sugar the
parser expanded. This is how many real languages implement compound operators, and
it shows a deep idea: new syntax is often cheapest to add as a transformation into
existing syntax, not as a new evaluator case.

</details>

**2. Detect uninitialized use at parse time (research + one language).** Right now
`z + 1` fails at *evaluation*. Could you catch an undefined variable earlier, while
parsing? What information would the parser need, and why do most languages still
defer this to a later pass?

<details>
<summary>Solution sketch</summary>

The parser alone can't know in general — a variable might be defined on an earlier
line it has already parsed, so you'd need to track a set of defined names as you
go (a tiny *semantic analysis* pass). Simple cases catch easily, but the general
problem (variables defined in other scopes, conditional definitions) is why real
compilers run a separate *resolution* or *name-analysis* pass between parsing and
evaluation/codegen. It's a preview of why the three-stage pipeline grows more
stages as a language grows.

</details>

**3. Add a `print` statement and block scope (any language).** Add `print expr`
so a program can output intermediate values, then add `{ ... }` blocks that get
their *own* environment layered over the outer one. What happens to a variable
assigned inside a block — and what does that tell you about how scopes nest?

<details>
<summary>Solution sketch</summary>

`print` is a new statement kind that evaluates its expression and emits it. Block
scope is the deeper part: a block gets a fresh environment whose *parent* is the
enclosing one, so a lookup that misses locally falls through to the parent — but
an assignment inside the block writes to the block's own layer, so it vanishes when
the block ends (unless you deliberately assign to an outer name). That parent-chain
of environments is *lexical scoping*, and it is exactly the structure functions and
closures will need — which is where the track goes next.

</details>

## Interview corner

<Interview question="A calculator evaluates expressions; a language has variables. Architecturally, what does it take to add them, and why is it a small change?">

It takes an **environment** — a runtime map from names to values — plus three
localized additions to the existing pipeline. The lexer learns to tokenize
identifiers and the assignment operator; the parser learns that an identifier is a
valid atomic expression (a variable reference) and that an identifier followed by
`=` begins an assignment statement rather than an expression; and the evaluator
threads the environment through, reading it on a variable reference and writing it
on an assignment. It's a small change precisely because the three-stage design has
clean seams: each stage gains one responsibility without disturbing the others.
The deep reason it matters is that the environment generalizes: give each function
call its own environment linked to the one it was defined in, and you have lexical
scoping and closures; stack those environments and you have the call stack. So
"add variables" is architecturally the moment a calculator becomes a language — one
map is the difference — and everything richer (functions, scopes, objects) is built
by giving that map more structure, not by rebuilding the interpreter.

</Interview>

## The track ahead

You now have a language with memory: a lexer, a parser producing statements and
expressions, and an evaluator with a persistent environment — built and tested
three ways, byte-identical. The next step is the big one: **functions**. A function
is a chunk of tree you can name and call, and each call needs its *own* environment
chained to the one where the function was defined — which is where closures and
lexical scoping come from, and where this humble `name → value` map reveals its
full power. From there: control flow, and a language that is unmistakably your own.
