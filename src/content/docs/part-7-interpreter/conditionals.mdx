<Badge text="Build Your Own X" variant="tip" size="large" />

Your language has [functions and closures](/systems-programming-academy/part-7-interpreter/functions/)
— but every program still runs straight to the end. It can't *decide*. This lesson
adds the one construct that changes that, and it is a bigger deal than it looks:
with comparisons and a conditional, **recursion can finally terminate**, and the
language crosses the line into **Turing-completeness** — it can now compute anything
computable.

```text
fact(n) = if n <= 1 then 1 else n * fact(n - 1)
fact(5)   # => 120
```

<Aside type="note" title="Why one branch, evaluated lazily, is the whole trick">
  The conditional here is an *expression*: `if cond then a else b`. Its defining
  property is that it evaluates the condition and then **only the taken branch** —
  never both. That laziness is exactly what lets `fact` stop: when `n <= 1`, the
  `else` branch (which would recurse again) is never evaluated. An eager `if` that
  computed both branches would recurse forever. Booleans are just integers here
  (`1` = true, `0` = false), which keeps every result exact and byte-identical.
</Aside>

## Comparisons: two-character tokens

Comparisons need operators the lexer hasn't seen: `<`, `<=`, `>`, `>=`, `==`, `!=`.
The new wrinkle is **two-character operators** — when the lexer sees `<`, it must
peek at the next character to tell `<` from `<=`. And `==` must be distinguished
from the single `=` of assignment.

```python title="cond.py"
nxt = src[i + 1] if i + 1 < len(src) else ""

    i += 2
elif c in "<>":
    tokens.append(Token("C", op=c))        # <, >
    i += 1
```

**Order matters:** always test the two-character form (`==`, `<=`) *before* the
one-character form (`=`, `<`), or the lexer would greedily grab `=` and leave a
stray `=` behind. This "longest match first" rule is universal in real lexers.

## The grammar grows two levels

The parser gains a comparison level (looser than `+`/`-`) and an `if` form at the
very top. The full precedence ladder is now:

```text
expr       := 'if' expr 'then' expr 'else' expr | comparison
comparison := add (('<'|'<='|'>'|'>='|'=='|'!=') add)?   ← loosest
add        := term (('+' | '-') term)*
term       := factor (('*' | '/') factor)*
factor     := Num | Ident '(' args ')' | Ident | '(' expr ')' | '-' factor
```

Because `comparison` sits below `add`, `n <= 1` parses as `(n) <= (1)` and
`a + b < c` as `(a + b) < c` — arithmetic binds tighter than comparison, exactly
as you'd expect. And `if` is the outermost form, so it can wrap any expression,
including another `if` (that's how `else if` chains work):

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="cond.py"
def expr(self) -> Expr:  # 'if' expr 'then' expr 'else' expr | comparison
    if self._is_kw("if"):
        self.pos += 1
        cond = self.expr()
        self._expect_kw("then")
        then = self.expr()
        self._expect_kw("else")
        els = self.expr()
        return Expr("I", cond=cond, then=then, els=els)
    return self.comparison()

def comparison(self) -> Expr:            # add (cmpop add)?
    left = self.add()
    t = self._peek()
    if t is not None and t.kind == "C":
        self.pos += 1
        return Expr("C", op=t.op, left=left, right=self.add())
    return left
```
</TabItem>
<TabItem label="Go">
```go title="cond.go"
// expr := 'if' expr 'then' expr 'else' expr | comparison
func (p *parser) expr() (*Expr, error) {
	if p.isKw("if") {
		p.pos++
		cond, _ := p.expr()
		if err := p.expectKw("then"); err != nil { return nil, err }
		then, _ := p.expr()
		if err := p.expectKw("else"); err != nil { return nil, err }
		els, _ := p.expr()
		return &Expr{Kind: 'I', Cond: cond, Then: then, Els: els}, nil
	}
	return p.comparison()
}

func (p *parser) comparison() (*Expr, error) {
	left, err := p.add()
	if err != nil { return nil, err }
	if t, ok := p.peek(); ok && t.Kind == 'C' {
		p.pos++
		right, err := p.add()
		if err != nil { return nil, err }
		return &Expr{Kind: 'C', Op: t.Op, L: left, R: right}, nil
	}
	return left, nil
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
// expr := 'if' expr 'then' expr 'else' expr | comparison
fn expr(&mut self) -> Result<Expr, String> {
    if self.is_kw("if") {
        self.pos += 1;
        let cond = self.expr()?;
        self.expect_kw("then")?;
        let then = self.expr()?;
        self.expect_kw("else")?;
        let els = self.expr()?;
        return Ok(Expr::If(Box::new(cond), Box::new(then), Box::new(els)));
    }
    self.comparison()
}

fn comparison(&mut self) -> Result<Expr, String> {
    let left = self.add()?;
    if let Some(Token::Cmp(op)) = self.peek() {
        let op = *op;
        self.pos += 1;
        return Ok(Expr::Cmp(op, Box::new(left), Box::new(self.add()?)));
    }
    Ok(left)
}
```
</TabItem>

`if`, `then`, and `else` are ordinary identifiers to the lexer; the parser just
treats them as keywords by name. It's the simplest possible approach and works
perfectly for a small language.

## Evaluation: compare, then take one branch

Two new `eval` cases. A comparison computes both sides and returns `1` or `0`. The
`if` computes the condition and then evaluates **only** the branch it selects —
the laziness that makes recursion possible:

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="cond.py"
if e.kind == "C":                                  # comparison
    a, b = _as_int(evaluate(e.left, env)), _as_int(evaluate(e.right, env))
    return 1 if _CMP[e.op](a, b) else 0            # booleans are integers
if e.kind == "I":                                  # if / then / else
    if _as_int(evaluate(e.cond, env)) != 0:
        return evaluate(e.then, env)               # ONLY the taken branch
    return evaluate(e.els, env)
```
</TabItem>

```rust title="src/lib.rs"
Expr::Cmp(op, l, r) => {
    let a = as_int(eval(l, env)?)?;
    let b = as_int(eval(r, env)?)?;
    let t = match *op { "<" => a < b, "<=" => a <= b, ">" => a > b,
                        ">=" => a >= b, "==" => a == b, "!=" => a != b, _ => unreachable!() };
    Ok(Value::Int(if t { 1 } else { 0 }))          // booleans are integers
}
Expr::If(cond, then, els) => {
    if as_int(eval(cond, env)?)? != 0 {
        eval(then, env)                            // ONLY the taken branch
    } else {
        eval(els, env)
    }
}
```
</TabItem>
</Tabs>

<Aside type="tip" title="Laziness is a correctness property, not an optimization">
  The test that proves it: `safe(n) = if n == 0 then 0 else 100 / n`. Calling
  `safe(0)` returns `0` with **no division-by-zero error** — because the `else`
  branch, `100 / 0`, is never evaluated. If the interpreter eagerly evaluated both
  branches (as it does for `+`'s operands), `safe(0)` would crash. Conditionals are
  the first place the language *must not* evaluate everything, and getting that
  right is what makes them useful.
</Aside>

## Recursion, now that it can stop

With a conditional, a recursive function has a base case to halt on. All three
implementations compute these identically:

```text
fact(n) = if n <= 1 then 1 else n * fact(n - 1)
fact(6)                                          => 720
fib(n)  = if n < 2 then n else fib(n - 1) + fib(n - 2)
fib(15)                                          => 610
gcd(a, b) = if b == 0 then a else gcd(b, a - a / b * b)   # Euclid's algorithm
gcd(48, 36)                                      => 12
```

That last one — Euclid's GCD, written in *your* language — is a quiet milestone: a
non-trivial algorithm running on an interpreter you built from a character scanner
up. **Functions plus conditionals is the threshold of Turing-completeness**: your
four-stage toy can now, in principle, compute anything any computer can.

## Exercises

**1. Add boolean `and` / `or` with short-circuit evaluation (all three).** `and`
and `or` must *not* evaluate their right side when the left already decides the
result (`false and X` is `false` without touching `X`). Why is short-circuiting the
same kind of laziness as `if`, and where does it belong in the precedence ladder?

<details>
<summary>Solution sketch</summary>

`and`/`or` are lazy in exactly the way `if` is: evaluate the left operand, and only
evaluate the right if the left didn't already settle the answer. They sit *below*
comparison in precedence (so `a < b and c < d` groups the comparisons first), and
they're implemented not as ordinary binary operators (which eagerly evaluate both
sides) but with the same branch-skipping the `if` case uses. That shared laziness is
why `x != 0 and 10 / x > 1` is safe — the division never runs when `x` is 0.

</details>

**2. Make recursion cheaper with tail calls (research + one language).** `fact` as
written builds a deep call stack. Rewrite it with an accumulator
(`fact(n, acc) = if n <= 1 then acc else fact(n - 1, n * acc)`) so the recursive
call is the *last* thing it does. What could an interpreter do with that shape that
it can't with the original?

<details>
<summary>Solution sketch</summary>

When the recursive call is in *tail position* — the very last operation, with
nothing left to do with its result — the interpreter can reuse the current call's
scope instead of nesting a new one, turning the recursion into a loop that runs in
constant stack space (tail-call optimization). The accumulator version has that
shape; the original `n * fact(n-1)` does not, because it still has to multiply after
the call returns. Recognizing tail calls is how functional languages recurse
arbitrarily deep without overflowing the stack.

</details>

**3. Add a `while`-style loop and compare it to recursion (any language).** Give the
language a loop construct, then implement `sum from 1 to n` both with a loop and with
recursion. With conditionals already present, is a loop adding any *power* to the
language — or only convenience?

<details>
<summary>Solution sketch</summary>

A loop adds no computational power: the language became Turing-complete the moment
it had recursion plus conditionals, and any loop can be rewritten as a recursive
function (and vice versa). What a loop adds is *convenience and efficiency* — an
explicit iteration that's easy to read and runs in constant stack space without
needing tail-call optimization. This is a deep point: many language features are
expressiveness and ergonomics layered on a small Turing-complete core, not new
capability. You have already built that core.

</details>

## Interview corner

<Interview question="Why must a conditional evaluate only one branch, and what breaks if it evaluates both?">

A conditional's whole purpose is *choice*, and choice implies not doing the thing
you didn't choose. Concretely, `if cond then a else b` must evaluate `cond`, then
evaluate exactly one of `a` or `b` — never both. Two things break if it evaluates
both. First, correctness: a branch may only be *valid* when it's selected — `if n ==
0 then 0 else 100 / n` divides by zero if the else branch runs when `n` is 0, so
eager evaluation would turn a safe program into a crashing one. Second, and more
fundamentally, termination: recursion relies on the recursive branch *not* running
once the base case is hit. `fact(n) = if n <= 1 then 1 else n * fact(n-1)` would
recurse forever if the interpreter evaluated the `else` branch even when `n <= 1`.
So laziness of the conditional isn't an optimization — it's what makes conditionals
mean anything at all, and it's the precise reason adding `if` to a language with
functions unlocks terminating recursion and, with it, Turing-completeness. This is
also why `and`/`or` short-circuit and why languages distinguish special forms (like
`if`) from ordinary functions (whose arguments are all evaluated): some constructs
*must* control what gets evaluated.

</Interview>

## The track ahead

Your language is now Turing-complete: a lexer, a recursive-descent parser, and a
tree-walking evaluator with variables, closures, and conditionals — everything
needed, in principle, to compute anything, built and tested three ways,
byte-identical. What's left is expressiveness and ergonomics, not power:
short-circuit booleans, `while` and `for` loops, `print` and strings, data
structures, better error messages with line numbers. Each is a modest addition to
the same four stages you now know intimately. You set out to demystify how a
language works — and you have built a real, if small, one from nothing but a
character scanner and these ideas.
