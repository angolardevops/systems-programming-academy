This part builds the tools that make the command line *feel* powerful — the
ones with color, live updating, and dashboards. We open with the flagship: a
**live system monitor**, a miniature `htop`, that reads the kernel's live
metrics and paints them as colored **heat bars** — green when calm, yellow when
warm, red when hot. It teaches two skills every elegant CLI needs: reading the
system's own state, and rendering a terminal UI from nothing but strings and
ANSI escape codes.

<Aside type="note" title="No magic, no privileges, no dependencies">
  Everything here reads Linux **`/proc`** — a virtual filesystem where the
  kernel exposes live stats as plain text files, readable by anyone. No root, no
  system-monitoring library, no C bindings. The whole monitor is text parsing
  plus string formatting, which is exactly why it stays testable and portable.
</Aside>

## The design that keeps a TUI testable

A live dashboard sounds untestable — it changes every second and paints the
screen. The trick is to split it into three pure pieces and only the outermost
one touches the terminal:

<Mermaid title="Parse (from text) -> compute (deltas) -> render (ANSI string); only main loops over live /proc" code={`flowchart LR
  P["/proc files (plain text)"] --> PA["parse: text -> numbers"]
  PA --> C["compute: deltas -> CPU%, rates"]
  C --> R["render: numbers -> ANSI frame string"]
  R --> M["main loop: read /proc, print frame, sleep, repeat"]
  M -.->|every second| P`} />

Parsing takes a *string*, so tests feed a fixed `/proc` snapshot instead of the
live kernel. Computing is pure arithmetic over two snapshots. Rendering returns
a *string* (ANSI codes and all), asserted byte-for-byte. Only `main` reads real
`/proc` in a loop and prints — the one part that isn't a pure function, and the
one part we don't unit-test. This is the same testability discipline as the
[mini-NGINX](/systems-programming-academy/part-4-concurrency/mini-nginx/) and the
[guestbook](/systems-programming-academy/capstone/guestbook/): push I/O to the
edge, keep the core pure.

## Reading the kernel: `/proc`

Each metric comes from a text file. CPU utilisation, for instance, is a *delta*:
`/proc/stat`'s `cpu` line counts jiffies spent in each state since boot, so you
sample it twice and divide the busy-time increase by the total increase.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="sysmon.py"
def parse_cpu(stat: str) -> CpuTimes:
    """Parse the aggregate `cpu` line of /proc/stat.
    Busy = everything except idle and iowait."""
    line = "cpu 0 0 0 0 0 0 0 0"
    for l in stat.splitlines():
        if l.startswith("cpu "):
            line = l
            break
    vals = [int(x) for x in line.split()[1:] if x.isdigit()]
    idle = _at(vals, 3) + _at(vals, 4)   # idle + iowait
    busy = sum(vals) - idle
    return CpuTimes(busy=busy, total=busy + idle)


def cpu_percent(prev: CpuTimes, cur: CpuTimes) -> float:
    total_delta = max(0, cur.total - prev.total)
    if total_delta == 0:
        return 0.0
    return max(0, cur.busy - prev.busy) * 100.0 / total_delta
```
</TabItem>
<TabItem label="Go">
```go title="sysmon.go"
func ParseCPU(stat string) CpuTimes {
	line := "cpu 0 0 0 0 0 0 0 0"
	for _, l := range strings.Split(stat, "\n") {
		if strings.HasPrefix(l, "cpu ") {
			line = l
			break
		}
	}
	fields := strings.Fields(line)[1:]
	var vals []uint64
	for _, f := range fields {
		if n, err := strconv.ParseUint(f, 10, 64); err == nil {
			vals = append(vals, n)
		}
	}
	idle := at(vals, 3) + at(vals, 4) // idle + iowait
	var sum uint64
	for _, v := range vals {
		sum += v
	}
	busy := sum - idle
	return CpuTimes{Busy: busy, Total: busy + idle}
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
pub fn parse_cpu(stat: &str) -> CpuTimes {
    let line = stat.lines().find(|l| l.starts_with("cpu "))
        .unwrap_or("cpu 0 0 0 0 0 0 0 0");
    let v: Vec<u64> = line.split_whitespace().skip(1)
        .filter_map(|n| n.parse().ok()).collect();
    let idle = v.get(3).copied().unwrap_or(0) + v.get(4).copied().unwrap_or(0);
    let busy: u64 = v.iter().sum::<u64>() - idle;
    CpuTimes { busy, total: busy + idle }
}
```
</TabItem>

Memory is a single snapshot (`MemTotal` and `MemAvailable` from `/proc/meminfo`);
network (`/proc/net/dev`) and disk (`/proc/diskstats`) are byte/sector counters
you turn into *rates* the same way as CPU — sample twice, divide the difference
by the interval. The tests pin every parser against a fixed snapshot: two CPU
samples give exactly 50%, the memory fixture gives exactly 70%, the network
fixture gives exactly 25 MB/s.

## Rendering heat bars with ANSI

The dashboard's look comes entirely from **ANSI escape codes** — the `\x1b[...m`
sequences terminals interpret as color and style. A heat bar is a run of filled
block characters (`█`) in a color chosen by the value, padded with dim blocks
(`░`):

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="sysmon.py"
def _heat_color(percent: float) -> int:
    if percent < 50.0:
        return 32   # green
    if percent < 80.0:
        return 33   # yellow
    return 31       # red


def render_bar(percent: float, width: int) -> str:
    pct = min(100.0, max(0.0, percent))
    filled = min(width, _round_half_up(pct / 100.0 * width))
    color = _heat_color(pct)
    return f"\x1b[{color}m{'█' * filled}{'░' * (width - filled)}\x1b[0m"
```
</TabItem>
<TabItem label="Go">
```go title="sysmon.go"
func heatColor(percent float64) int {
	switch {
	case percent < 50.0:
		return 32 // green
	case percent < 80.0:
		return 33 // yellow
	default:
		return 31 // red
	}
}

func RenderBar(percent float64, width int) string {
	pct := clamp(percent, 0, 100)
	filled := roundHalfUp(pct / 100.0 * float64(width))
	color := heatColor(pct)
	return fmt.Sprintf("\x1b[%dm%s%s\x1b[0m",
		color, strings.Repeat("█", filled), strings.Repeat("░", width-filled))
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
fn heat_color(percent: f64) -> u8 {
    if percent < 50.0 { 32 }        // green
    else if percent < 80.0 { 33 }   // yellow
    else { 31 }                     // red
}

pub fn render_bar(percent: f64, width: usize) -> String {
    let pct = percent.clamp(0.0, 100.0);
    let filled = round_half_up(pct / 100.0 * width as f64).min(width);
    let color = heat_color(pct);
    format!("\x1b[{color}m{}{}\x1b[0m",
        "█".repeat(filled), "░".repeat(width - filled))
}
```
</TabItem>
</Tabs>

`\x1b[33m` starts yellow text, `\x1b[0m` resets — everything between is colored.
`32`/`33`/`31` are green/yellow/red. The bar's *fill* is the value scaled to the
bar width, and getting that byte-identical across three languages meant one
subtle fix: **rounding.** Rust's `round()`, Go's `math.Round`, and Python's
`round()` don't all break ties the same way (Python uses banker's rounding), so
the code uses an explicit `floor(x + 0.5)` everywhere — the same rounding in all
three, so `50%` of a 20-cell bar is always exactly 10 filled cells.

<Aside type="tip" title="The rounding trap is real">
  A dashboard that renders one cell differently in Python than in Rust would be
  a mystery bug born entirely of each language's default `round()`. Any time you
  need identical numeric output across languages, pin the rounding explicitly —
  it is the float-formatting lesson from the
  [serialization](/systems-programming-academy/part-5-frameworks/serialization/)
  project, striking again.
</Aside>

## Making it live

The only impure part is `main`: read `/proc`, print the frame, sleep, repeat —
and two escape codes turn a scrolling program into a stable dashboard.
`\x1b[2J` clears the screen once; `\x1b[H` moves the cursor **home** (top-left)
before each redraw, so each frame overwrites the last instead of scrolling:

```rust title="src/main.rs — the live loop"
print!("\x1b[2J");                 // clear once
loop {
    sleep(Duration::from_secs_f64(1.0));
    // ... read /proc, compute cpu/mem/net/disk ...
    print!("\x1b[H");              // cursor home — redraw in place
    println!("{}", render_dashboard(cpu, mem, net, disk_bps));
    // ... carry current readings forward as "prev" ...
}
```

Run it (`cargo run --release`, `go run .`, or `python3 sysmon.py`) and you have
a real, live monitor — reading your actual machine, no privileges required:

```text
 CPU  ██░░░░░░░░░░░░░░░░░░  9.5%       (green)
 MEM  ███████████░░░░░░░░░  57.0%      (yellow)
```

is made of.

## Exercises

**1. Add a sparkline history (all three).** Keep the last N CPU readings and
render them as a row of block characters of varying height (`▁▂▃▄▅▆▇█`) — a
tiny inline chart. What has to move from `render_bar` into shared state, and why
does the renderer stay a pure function of that state?

<details>
<summary>Solution sketch</summary>

The history (a ring buffer of the last N percentages) becomes state the `main`
loop owns and passes into a new `render_sparkline(history)` — which is still a
pure function: same history in, same string out. The block-height character is
`floor(pct/100 * 7)` indexed into `▁…█`. Keeping the renderer pure means you can
test it against a fixed history, exactly as you tested the bars. State lives in
the loop; rendering stays a function of it.

</details>

**2. Auto-scale the network bar (any language).** The fixed 1 Gbit reference
makes a busy 100 Mbit link look empty. Track the max rate seen and scale the bar
against it, so the bar always uses the full width. What's the trade-off versus a
fixed scale, for reading the dashboard at a glance?

<details>
<summary>Solution sketch</summary>

Auto-scaling (bar = current / max-seen) always fills the width, showing *relative*
activity well, but the absolute meaning shifts as the max grows — a full bar
means "the busiest it's been", not "saturated". A fixed scale shows *absolute*
load consistently but wastes width on slow links. Real monitors often show both:
a fixed-reference color and an auto-scaled bar. The choice is relative-trend vs
absolute-level, and knowing which your reader needs is the design decision.

</details>

**3. Add per-core CPU (Rust or Go).** `/proc/stat` has `cpu0`, `cpu1`, … lines
too. Parse and render a bar per core. What changes in the parser, and how do you
keep the frame deterministic when core count varies by machine?

<details>
<summary>Solution sketch</summary>

The parser returns a list of `CpuTimes` (aggregate plus per-core), and the
renderer emits one bar per core. Determinism-across-machines is a testing
concern: unit-test the render against a *fixed* set of core samples (so the
frame is stable in CI), while the live binary adapts to the real core count.
This is the same split as everywhere in the project — fixtures for the tested
core, live data for the running tool.

</details>

## Interview corner

<Interview question="How does a tool like top or htop get CPU, memory, and I/O without special privileges?">

On Linux, from the `/proc` virtual filesystem — a kernel-backed, world-readable
tree of plain-text files exposing live system state. CPU comes from
`/proc/stat` (cumulative jiffies per state; utilisation is the delta between two
reads), memory from `/proc/meminfo` (a snapshot of totals and available),
network from `/proc/net/dev` (per-interface byte counters, turned into rates by
sampling), disk from `/proc/diskstats` (sector counters, likewise). No syscalls
beyond `open`/`read`, no root, no library — which is why a monitor is portable
and easy to build. The one recurring pattern is that most of these are
*counters since boot*, so every "current" number is really a rate: sample twice,
divide by the interval. (Per-process stats live under `/proc/<pid>/`, same
idea.)

</Interview>

<Interview question="How do you build a terminal UI that updates in place instead of scrolling?">

With ANSI escape sequences — control codes the terminal interprets rather than
prints. The essentials are `\x1b[2J` to clear the screen and `\x1b[H` to move the
cursor to the home position (top-left); by printing `\x1b[H` before each redraw,
every frame overwrites the previous one in place, producing a stable dashboard
instead of a scroll. Color comes from SGR codes (`\x1b[32m` green, `\x1b[0m`
reset). For richer UIs you add cursor save/restore, line clearing (`\x1b[K`), and
alternate-screen mode (`\x1b[?1049h`, so quitting restores the shell). Libraries
like ncurses, crossterm, or bubbletea wrap all this, but it is just bytes on
stdout — building a monitor by hand, as here, demystifies every TUI you will ever
use. The discipline is to keep the *rendering* a pure function that returns the
frame as a string, and let only the loop emit the control codes.

</Interview>
