A [lição de routing](/systems-programming-academy/pt/part-5-frameworks/routing-middleware/)
construiu uma framework à volta do fluxo de controlo. Esta constrói uma framework
à volta de *construir uma string em segurança* — o que soa trivial até perceberes
que a vulnerabilidade web mais catastrófica e mais comum da história é
exatamente uma string construída mal. Construímos um **query builder SQL**: uma
API fluente e encadeável que transforma chamadas de método em SQL.

Duas ideias carregam a lição. O **padrão builder** — cada método devolve o
builder, para que as chamadas se encadeiem numa frase legível. E as **queries
parametrizadas** — os valores do utilizador nunca tocam no texto SQL; tornam-se
placeholders `?` e viajam numa lista separada. Essa segunda ideia não é um
adorno. É a defesa estrutural que torna a SQL injection *impossível de exprimir*
em código que usa o builder.

<Aside type="note" title="Porque é isto uma preocupação de framework e não só um helper">
  ORMs e query builders (SQLAlchemy, Diesel, GORM, activerecord) existem em
  grande parte porque escrever SQL à mão com concatenação de strings é
  catastroficamente fácil de errar. Ao tornar o caminho *seguro* no *único*
  caminho — não consegues literalmente pôr um valor de utilizador em lado nenhum
  a não ser na lista de params — a framework remove uma classe inteira de
  vulnerabilidades por construção. É a forma mais alta de design de API: tornar a
  coisa errada irrepresentável.
</Aside>

## A API fluente

O objetivo é que uma query se leia quase como a frase que dirias em voz alta:

```text
Query::table("orders")
    .select(&["id", "total"])
    .where_("status", "=", "paid")
    .order_by("total")
    .limit(10)
    .build()
```

compila para:

```sql
SELECT id, total FROM orders WHERE status = ? ORDER BY total LIMIT 10
-- params: ["paid"]
```

Cada método devolve o builder para a chamada seguinte se encadear — esse
devolver-self *é* o padrão builder. O `build()` percorre o estado acumulado uma
vez e emite o SQL mais os params posicionais.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="querybuilder.py"
def where(self, column: str, op: str, value: str) -> Query:
    """Adiciona uma condição ``column op ?``, ligando ``value`` como
    parâmetro. Múltiplas chamadas são juntadas com ``AND`` pela ordem."""
    self._conditions.append(_Condition(column, op, value))
    return self
```

O Python devolve `self`, o análogo direto da abordagem por ponteiro do Go. A
type hint `-> Query` é o que faz os editores autocompletar o método seguinte da
cadeia — uma coisa pequena que torna uma API fluente agradável de usar.
</TabItem>
<TabItem label="Go">
```go title="query.go"
// Where adiciona uma condição "column op ?", ligando value como parâmetro.
// Múltiplas chamadas são juntadas com AND pela ordem de chamada.
func (q *Query) Where(column, op, value string) *Query {
	q.conditions = append(q.conditions, condition{column, op, value})
	return q
}
```

O Go devolve o ponteiro `*Query`, mutando no sítio e devolvendo o mesmo builder.
Idiomático e sem alocação por chamada, ao custo de o builder ser estado mutável
partilhado (bem aqui — constróis uma query, uma vez).
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
/// Adiciona uma condição `column op ?`, ligando `value` como parâmetro.
/// Múltiplas chamadas são juntadas com `AND` pela ordem de chamada.
pub fn where_(mut self, column: &str, op: &str, value: &str) -> Self {
    self.conditions.push(Condition {
        column: column.to_string(),
        op: op.to_string(),
        value: value.to_string(),
    });
    self
}
```

O Rust recebe `self` por valor e devolve-o (`mut self -> Self`), por isso a
cadeia *move* o builder ao longo — sem borrows para gerir, e o compilador garante
que o builder é usado linearmente. (`where` é keyword, daí `where_`.)
</TabItem>
</Tabs>

## A regra única que importa: os valores viram `?`

Eis o argumento de segurança inteiro num só método. Quando acrescentas um
`WHERE`, o valor **não** é escrito no SQL. Um placeholder `?` vai para o SQL; o
valor vai para uma lista de params separada.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="querybuilder.py"
if self._conditions:
    clauses = []
    for c in self._conditions:
        params.append(c.value)  # valor -> params
        clauses.append(f"{c.column} {c.op} ?")  # SQL recebe só '?'
    sql += " WHERE " + " AND ".join(clauses)
```
</TabItem>
<TabItem label="Go">
```go title="query.go"
if len(q.conditions) > 0 {
	clauses := make([]string, len(q.conditions))
	for i, c := range q.conditions {
		params = append(params, c.value) // valor -> params
		clauses[i] = fmt.Sprintf("%s %s ?", c.column, c.op) // SQL recebe '?'
	}
	sql += " WHERE " + strings.Join(clauses, " AND ")
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
if !self.conditions.is_empty() {
    let clauses: Vec<String> = self
        .conditions
        .iter()
        .map(|c| {
            params.push(c.value.clone()); // valor -> params
            format!("{} {} ?", c.column, c.op) // SQL recebe só '?'
        })
        .collect();
    sql.push_str(" WHERE ");
    sql.push_str(&clauses.join(" AND "));
}
```
</TabItem>
</Tabs>

No momento da execução o driver da base de dados recebe o SQL e os params
*separadamente* e liga-os ele próprio — o valor nunca é parseado como SQL. A
coluna e o operador são estruturais e vêm do teu código, não do utilizador; só o
valor é controlado pelo utilizador, e o valor só pode ser um parâmetro ligado.

## Prova: a tentativa de injeção é inerte

Cada implementação tem o mesmo teste a disparar a string de ataque canónica
contra um slot de valor:

```text
value = "'; DROP TABLE users; --"

produz:
  SQL:    SELECT * FROM users WHERE name = ?
  params: ["'; DROP TABLE users; --"]
```

O payload aterra em `params` como dados inertes. O SQL mantém o seu único `?`, e
o `sql.contains("DROP")` é verificado como falso. Compara com a versão ingénua
contra a qual isto defende:

```python
# NUNCA faças isto — a vulnerabilidade clássica
sql = f"SELECT * FROM users WHERE name = '{value}'"
# torna-se: SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
```

<Mermaid title="A interpolacao de string deixa o valor tornar-se codigo; um placeholder mantem-no dados" code={`flowchart TD
  V["valor do utilizador: '; DROP TABLE users; --"] --> C{como e combinado?}
  C -->|interpolacao de string| BAD["valor parseado COMO SQL -> tabela apagada"]
  C -->|placeholder + param| SAFE["valor ligado como DADOS -> string literal comparada"]
  BAD --> X["injecao"]
  SAFE --> OK["seguro por construcao"]`} />

Isto é o [Bobby Tables](https://xkcd.com/327/) tornado concreto. A versão
interpolada trata a string do utilizador como sintaxe SQL; a versão parametrizada
só pode tratá-la como um valor a comparar. O mesmo input, dois universos.

## SQL byte-idêntico nas três linguagens

Como em todos os projetos multi-linguagem desta academia, os três builders foram
corridos na mesma query e o output comparado:

```text
Rust:   SELECT id, name, email FROM users WHERE age >= ? AND country = ? ORDER BY name LIMIT 25
Go:     SELECT id, name, email FROM users WHERE age >= ? AND country = ? ORDER BY name LIMIT 25
Python: SELECT id, name, email FROM users WHERE age >= ? AND country = ? ORDER BY name LIMIT 25
```

Idêntico ao byte. Essa concordância é o que deixa os *testes* serem partilhados:
as mesmas sete asserções (`SELECT * FROM users`, o juntar com AND, o payload de
injeção a ficar em params) passam tal e qual nas três, o que só é significativo
porque as implementações concordam exatamente sobre que SQL emitir.

<Aside type="tip" title="Onde os builders reais vão mais longe">
  Builders de produção acrescentam: quoting de identificadores (para uma coluna
  chamada `order` não partir), diferenças de dialeto (`LIMIT` vs `TOP` vs
  `FETCH FIRST`), joins, sub-queries, e nomes de coluna verificados em tempo de
  compilação (Diesel, sqlx). Mas todos eles mantêm o núcleo inegociável desta
  lição: os valores são parâmetros, nunca interpolados. O resto são
  funcionalidades; isto é segurança.
</Aside>

## Exercícios

**1. Acrescenta `insert` (nas três).** Estende o builder com
`table("users").insert({"name": ..., "age": ...})` a produzir
`INSERT INTO users (name, age) VALUES (?, ?)` com os valores como params. Atenção
à ordem: a ordem das colunas e a ordem dos valores têm de corresponder
exatamente. Como os manténs em sincronia em cada linguagem?

<details>
<summary>Esboço de solução</summary>

Itera as colunas e os valores juntos a partir de uma estrutura ordenada para não
poderem divergir — em Rust um `Vec<(String, String)>`, em Go uma slice de pares,
em Python um dict ordenado (ordem de inserção desde 3.7). A armadilha é usar um
mapa desordenado para as colunas e uma lista separada para os valores; eles *vão*
desalinhar. Manter o par junto é a mesma disciplina que mantém o `?` e o seu
param alinhados na cláusula WHERE.

</details>

**2. Suporta `OR` e agrupamento (qualquer linguagem).** Acrescenta `or_where(...)`
e raciocina sobre precedência: `a AND b OR c` não é `a AND (b OR c)`. Como deixas
quem chama exprimir condições agrupadas sem os deixar injetar SQL cru?

<details>
<summary>Esboço de solução</summary>

Modela as condições como uma pequena árvore (nós AND/OR com folhas de condição)
em vez de uma lista plana, e emite parênteses ao serializar um grupo aninhado.
Quem chama constrói a árvore através de métodos — nunca passando uma string de
cláusula crua — por isso a invariante de segurança-contra-injeção aguenta: a
estrutura vem de chamadas de método, os valores de params. É o ponto onde um
builder começa a tornar-se um verdadeiro compilador de expressões.

</details>

**3. Valida identificadores (nas três).** Um nome de tabela ou coluna *é*
interpolado (não pode ser um `?`), por isso um nome de coluna malicioso é um vetor
de injeção real. Acrescenta validação que rejeita qualquer identificador que não
case com `^[A-Za-z_][A-Za-z0-9_]*$`. Porque é que uma regex de allowlist é a
ferramenta certa aqui e uma denylist a errada?

<details>
<summary>Esboço de solução</summary>

Os identificadores não podem ser parametrizados, por isso têm de ser *validados*
— e a validação tem de ser uma allowlist (só caracteres conhecidos como seguros
permitidos), nunca uma denylist (tentar enumerar caracteres maus, que falha
sempre alguma coisa). A regex `^[A-Za-z_][A-Za-z0-9_]*$` admite só identificadores
simples, rejeitando aspas, espaços, ponto-e-vírgulas e marcadores de comentário
de imediato. Isto fecha a única porta de injeção que a parametrização não pode: as
partes estruturais da query que têm de vir do código, verificadas a parecer mesmo
código.

</details>

## Cantinho da entrevista

<Interview question="Explica SQL injection e a forma correta de a prevenir.">

A SQL injection acontece quando dados controlados pelo utilizador são combinados
com uma query SQL de uma forma que deixa os dados mudar a *estrutura* da query —
classicamente construindo o SQL com concatenação de strings, para que um valor
como `'; DROP TABLE users; --` se torne sintaxe executável. A prevenção correta e
robusta são os prepared statements parametrizados: o texto da query com
placeholders `?` (ou `$1`) é enviado à base de dados separado dos valores, e o
driver liga os valores como dados que nunca são parseados como SQL. Crucialmente
isto não é escaping — o escaping é um fallback mais fraco e propenso a erros. Para
as partes que genuinamente não podem ser parametrizadas, nomeadamente
identificadores como nomes de tabela e coluna, validas contra uma allowlist. Um
query builder ou ORM impõe tudo isto por construção, e é por isso que "usa queries
parametrizadas, sempre" é a resposta de uma linha e "construímos SQL com f-strings"
é uma red flag.

</Interview>

<Interview question="O que é o padrão builder, e quando é a escolha certa em vez de um construtor ou um struct de config?">

O padrão builder constrói um objeto passo a passo através de métodos encadeados,
cada um devolvendo o builder, terminando num `build()` que produz o valor final.
Brilha quando um objeto tem muitos parâmetros opcionais — um construtor com dez
argumentos, a maioria com defaults, é ilegível e propenso a erros, enquanto
`.select(...).where(...).limit(...)` nomeia cada escolha no local da chamada e
deixa-te omitir o que não precisas. Também permite validação ou transformação no
momento do `build()`, e pode impor distinções obrigatório-vs-opcional (um builder
type-state não compila sem as peças obrigatórias). O trade-off é mais código do
que um struct simples e um momento em que o builder está num estado parcialmente
construído e inválido — por isso para um valor com dois ou três campos
obrigatórios, um construtor é mais simples e melhor. A construção de queries é o
encaixe de manual: cláusulas opcionais abertas, montadas fluentemente.

</Interview>

<Interview question="Onde é que query builders e ORMs ainda te deixam exposto, e como cobres isso?">

Fecham a porta da injeção de valores mas não todas as portas. Identificadores
encaminhados de input do utilizador (`ORDER BY {col}` dinâmico, nomes de tabela
de tenant) são interpolados, não parametrizados, por isso precisam de validação
por allowlist — um vetor real e muitas vezes esquecido. Escape hatches de SQL cru
(`.raw()`, `text()`) devolvem-te a arma carregada para os casos que o builder não
consegue exprimir, e são onde a injeção volta a insinuar-se; exigem a mesma
disciplina de parametrização à mão. Há também o `LIMIT` / `OFFSET` e o binding de
listas `IN (...)`, que alguns drivers tratam de forma desajeitada. E nada disto
toca na autorização: uma query perfeitamente parametrizada pode ainda ler linhas
que o utilizador não devia ver se esqueceres o `WHERE tenant_id = ?`. O builder
torna o caso comum seguro; o risco residual são exatamente as partes que saem
dele, que é onde a revisão e a validação se focam.

</Interview>

## Desafio

Transforma isto numa pequena camada de dados type-safe: define um struct/classe
`User`, gera `SELECT`, `INSERT`, `UPDATE` e `DELETE` para ele, acrescenta
validação de identificadores, e liga-o a uma base de dados SQLite real (stdlib em
Python, `database/sql` em Go, o crate `rusqlite` em Rust) para os pares
`(sql, params)` executarem mesmo. Escreve um teste de integração que insere uma
linha cujo nome é o payload Bobby Tables, o lê de volta intacto, e confirma que a
tabela ainda existe — prova de que a injeção é neutralizada de ponta a ponta, não
só na string gerada. Terás construído o núcleo seguro que todo o ORM embrulha em
funcionalidades, e nunca mais serás tentado a pegar numa f-string numa query.
