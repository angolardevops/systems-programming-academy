A tua linguagem tem [funções e closures](/systems-programming-academy/pt/part-7-interpreter/functions/)
— mas todo o programa ainda corre direito até ao fim. Não consegue *decidir*. Esta
lição adiciona a única construção que muda isso, e é mais importante do que parece:
com comparações e um condicional, **a recursão pode finalmente terminar**, e a
linguagem atravessa a linha da **Turing-completude** — passa a poder computar tudo o
que é computável.

```text
fact(n) = if n <= 1 then 1 else n * fact(n - 1)
fact(5)   # => 120
```

<Aside type="note" title="Porque um ramo, avaliado preguiçosamente, é o truque todo">
  O condicional aqui é uma *expressão*: `if cond then a else b`. A sua propriedade
  definidora é que avalia a condição e depois **só o ramo tomado** — nunca ambos.
  Essa preguiça é exatamente o que deixa o `fact` parar: quando `n <= 1`, o ramo
  `else` (que voltaria a recorrer) nunca é avaliado. Um `if` ansioso que computasse
  ambos os ramos recorreria para sempre. Os booleanos são só inteiros aqui (`1` =
  verdadeiro, `0` = falso), o que mantém cada resultado exato e byte-idêntico.
</Aside>

## Comparações: tokens de dois caracteres

As comparações precisam de operadores que o lexer ainda não viu: `<`, `<=`, `>`,
`>=`, `==`, `!=`. A ruga nova são os **operadores de dois caracteres** — quando o
lexer vê `<`, tem de espreitar o caractere seguinte para distinguir `<` de `<=`. E o
`==` tem de ser distinguido do `=` único da atribuição.

**A ordem importa:** testa sempre a forma de dois caracteres (`==`, `<=`) *antes* da
forma de um caractere (`=`, `<`), ou o lexer agarraria gananciosamente o `=` e
deixaria um `=` perdido para trás. Esta regra "match mais longo primeiro" é universal
em lexers reais.

## A gramática cresce dois níveis

O parser ganha um nível de comparação (mais frouxo que `+`/`-`) e uma forma `if` no
topo. A escada de precedência completa é agora:

```text
expr       := 'if' expr 'then' expr 'else' expr | comparison
comparison := add (('<'|'<='|'>'|'>='|'=='|'!=') add)?   ← mais frouxo
add        := term (('+' | '-') term)*
term       := factor (('*' | '/') factor)*
factor     := Num | Ident '(' args ')' | Ident | '(' expr ')' | '-' factor
```

Como o `comparison` fica abaixo do `add`, `n <= 1` faz parse como `(n) <= (1)` e
`a + b < c` como `(a + b) < c` — a aritmética liga mais forte que a comparação,
exatamente como esperarias. E o `if` é a forma mais exterior, então pode envolver
qualquer expressão, incluindo outro `if` (é assim que cadeias `else if` funcionam):

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="cond.py"
def expr(self) -> Expr:  # 'if' expr 'then' expr 'else' expr | comparison
    if self._is_kw("if"):
        self.pos += 1
        cond = self.expr()
        self._expect_kw("then")
        then = self.expr()
        self._expect_kw("else")
        els = self.expr()
        return Expr("I", cond=cond, then=then, els=els)
    return self.comparison()

def comparison(self) -> Expr:            # add (cmpop add)?
    left = self.add()
    t = self._peek()
    if t is not None and t.kind == "C":
        self.pos += 1
        return Expr("C", op=t.op, left=left, right=self.add())
    return left
```
</TabItem>

```go title="cond.go"
// expr := 'if' expr 'then' expr 'else' expr | comparison
func (p *parser) expr() (*Expr, error) {
	if p.isKw("if") {
		p.pos++
		cond, _ := p.expr()
		if err := p.expectKw("then"); err != nil { return nil, err }
		then, _ := p.expr()
		if err := p.expectKw("else"); err != nil { return nil, err }
		els, _ := p.expr()
		return &Expr{Kind: 'I', Cond: cond, Then: then, Els: els}, nil
	}
	return p.comparison()
}

func (p *parser) comparison() (*Expr, error) {
	left, err := p.add()
	if err != nil { return nil, err }
	if t, ok := p.peek(); ok && t.Kind == 'C' {
		p.pos++
		right, err := p.add()
		if err != nil { return nil, err }
		return &Expr{Kind: 'C', Op: t.Op, L: left, R: right}, nil
	}
	return left, nil
}
```
</TabItem>

```rust title="src/lib.rs"
// expr := 'if' expr 'then' expr 'else' expr | comparison
fn expr(&mut self) -> Result<Expr, String> {
    if self.is_kw("if") {
        self.pos += 1;
        let cond = self.expr()?;
        self.expect_kw("then")?;
        let then = self.expr()?;
        self.expect_kw("else")?;
        let els = self.expr()?;
        return Ok(Expr::If(Box::new(cond), Box::new(then), Box::new(els)));
    }
    self.comparison()
}

fn comparison(&mut self) -> Result<Expr, String> {
    let left = self.add()?;
    if let Some(Token::Cmp(op)) = self.peek() {
        let op = *op;
        self.pos += 1;
        return Ok(Expr::Cmp(op, Box::new(left), Box::new(self.add()?)));
    }
    Ok(left)
}
```
</TabItem>

O `if`, o `then` e o `else` são identificadores comuns para o lexer; o parser apenas
os trata como keywords por nome. É a abordagem mais simples possível e funciona
perfeitamente para uma linguagem pequena.

## Avaliação: compara, depois toma um ramo

Dois casos novos de `eval`. Uma comparação computa ambos os lados e devolve `1` ou
`0`. O `if` computa a condição e depois avalia **só** o ramo que seleciona — a
preguiça que torna a recursão possível:

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="cond.py"

    a, b = _as_int(evaluate(e.left, env)), _as_int(evaluate(e.right, env))
    return 1 if _CMP[e.op](a, b) else 0            # booleanos são inteiros
if e.kind == "I":                                  # if / then / else
    if _as_int(evaluate(e.cond, env)) != 0:
        return evaluate(e.then, env)               # SÓ o ramo tomado
    return evaluate(e.els, env)
```
</TabItem>

```rust title="src/lib.rs"
Expr::Cmp(op, l, r) => {
    let a = as_int(eval(l, env)?)?;
    let b = as_int(eval(r, env)?)?;
    let t = match *op { "<" => a < b, "<=" => a <= b, ">" => a > b,
                        ">=" => a >= b, "==" => a == b, "!=" => a != b, _ => unreachable!() };
    Ok(Value::Int(if t { 1 } else { 0 }))          // booleanos são inteiros
}
Expr::If(cond, then, els) => {
    if as_int(eval(cond, env)?)? != 0 {
        eval(then, env)                            // SÓ o ramo tomado
    } else {
        eval(els, env)
    }
}
```

<Aside type="tip" title="A preguiça é uma propriedade de correção, não uma otimização">
  O teste que o prova: `safe(n) = if n == 0 then 0 else 100 / n`. Chamar `safe(0)`
  devolve `0` com **nenhum erro de divisão por zero** — porque o ramo `else`,
  `100 / 0`, nunca é avaliado. Se o interpretador avaliasse ansiosamente ambos os
  ramos (como faz para os operandos do `+`), `safe(0)` rebentaria. Os condicionais
  são o primeiro sítio onde a linguagem *não pode* avaliar tudo, e acertar nisso é o
  que os torna úteis.
</Aside>

## Recursão, agora que pode parar

Com um condicional, uma função recursiva tem um caso-base onde parar. As três
implementações computam estes de forma idêntica:

```text
fact(n) = if n <= 1 then 1 else n * fact(n - 1)
fact(6)                                          => 720
fib(n)  = if n < 2 then n else fib(n - 1) + fib(n - 2)
fib(15)                                          => 610
gcd(a, b) = if b == 0 then a else gcd(b, a - a / b * b)   # algoritmo de Euclides
gcd(48, 36)                                      => 12
```

Esse último — o GCD de Euclides, escrito na *tua* linguagem — é um marco silencioso:
um algoritmo não-trivial a correr num interpretador que construíste de um scanner de
caracteres para cima. **Funções mais condicionais é o limiar da Turing-completude**:
o teu brinquedo de quatro fases pode agora, em princípio, computar tudo o que
qualquer computador pode.

CMPF = {"<": lambda a, b: a < b, "<=": lambda a, b: a <= b, ">": lambda a, b: a > b,
        ">=": lambda a, b: a >= b, "==": lambda a, b: a == b, "!=": lambda a, b: a != b}

def tokenize(s):
    toks, i = [], 0
    while i < len(s):
        c = s[i]; nx = s[i + 1] if i + 1 < len(s) else ""
        if c in " \\t": i += 1
        elif c == "=" and nx == "=": toks.append("=="); i += 2
        elif c in "!<>" and nx == "=": toks.append(c + "="); i += 2
        elif c in "<>": toks.append(c); i += 1
        elif c in "=,+-*/()": toks.append(c); i += 1
        elif c.isdigit():
            j = i
            while j < len(s) and s[j].isdigit(): j += 1
            toks.append(int(s[i:j])); i = j
        elif c.isalpha() or c == "_":
            j = i
            while j < len(s) and (s[j].isalnum() or s[j] == "_"): j += 1
            toks.append(("id", s[i:j])); i = j
        else: raise ValueError(f"unexpected character {c!r}")
    return toks

class P:
    def __init__(s, t): s.t, s.i = t, 0
    def peek(s): return s.t[s.i] if s.i < len(s.t) else None
    def nxt(s):
        v = s.peek(); s.i += 1; return v
    def iskw(s, k): return s.peek() == ("id", k)
    def expr(s):
        if s.iskw("if"):
            s.nxt(); cond = s.expr()
            if not s.iskw("then"): raise ValueError("expected 'then'")
            s.nxt(); then = s.expr()
            if not s.iskw("else"): raise ValueError("expected 'else'")
            s.nxt(); els = s.expr()
            return ("if", cond, then, els)
        return s.cmp()
    def cmp(s):
        left = s.add()
        if s.peek() in CMP:
            op = s.nxt(); return ("cmp", op, left, s.add())
        return left
    def add(s):
        n = s.term()
        while s.peek() in ("+", "-"):
            op = s.nxt(); n = (op, n, s.term())
        return n
    def term(s):
        n = s.factor()
        while s.peek() in ("*", "/"):
            op = s.nxt(); n = (op, n, s.factor())
        return n
    def factor(s):
        t = s.nxt()
        if isinstance(t, int): return t
        if isinstance(t, tuple):
            if s.peek() == "(":
                s.nxt(); args = []
                if s.peek() != ")":
                    args.append(s.expr())
                    while s.peek() == ",": s.nxt(); args.append(s.expr())
                if s.nxt() != ")": raise ValueError("expected ')'")
                return ("call", t[1], args)
            return ("var", t[1])
        if t == "-": return ("neg", s.factor())
        if t == "(":
            n = s.expr()
            if s.nxt() != ")": raise ValueError("expected ')'")
            return n
        raise ValueError("unexpected end of input" if t is None else f"unexpected {t!r}")

class Scope:
    def __init__(s, parent=None): s.v, s.parent = {}, parent
    def get(s, n):
        sc = s
        while sc:
            if n in sc.v: return sc.v[n]
            sc = sc.parent
        return None

def asint(v):
    if isinstance(v, int): return v
    raise ValueError("cannot do arithmetic on a function")

def ev(n, env):
    if isinstance(n, int): return n
    k = n[0]
    if k == "var":
        v = env.get(n[1])
        if v is None: raise ValueError(f"undefined variable {n[1]!r}")
        return v
    if k == "neg": return -asint(ev(n[1], env))
    if k == "cmp":
        a, b = asint(ev(n[2], env)), asint(ev(n[3], env))
        return 1 if CMPF[n[1]](a, b) else 0
    if k == "if":
        return ev(n[2], env) if asint(ev(n[1], env)) != 0 else ev(n[3], env)
    if k == "call":
        fn = env.get(n[1])
        if not (isinstance(fn, tuple) and fn[0] == "fn"):
            raise ValueError(f"{n[1]!r} is not a function")
        _, params, body, defenv = fn
        args = [ev(a, env) for a in n[2]]
        if len(args) != len(params): raise ValueError(f"{n[1]!r} arity mismatch")
        call = Scope(defenv)
        for p, a in zip(params, args): call.v[p] = a
        return ev(body, call)
    a, b = asint(ev(n[1], env)), asint(ev(n[2], env))
    if k == "+": return a + b
    if k == "-": return a - b
    if k == "*": return a * b
    if b == 0: raise ValueError("division by zero")
    q = abs(a) // abs(b)
    return -q if (a < 0) != (b < 0) else q

def params_of(toks):
    ps, i = [], 0
    if toks[i] == ")": return ps, i + 1
    while True:
        ps.append(toks[i][1]); i += 1
        if toks[i] == ",": i += 1; continue
        if toks[i] == ")": return ps, i + 1
        raise ValueError("bad parameter list")

def run(program):
    env = Scope()
    for line in program.strip().split("\\n"):
        line = line.strip()
        if not line: continue
        try:
            t = tokenize(line)
            is_def = len(t) > 2 and isinstance(t[0], tuple) and t[1] == "(" and ")" in t and t[t.index(")") + 1:t.index(")") + 2] == ["="]

                print(f"{line:<48} => {r if isinstance(r, int) else '<fn>'}")
        except ValueError as e:
            print(f"{line:<48} -> error: {e}")

## Exercícios

**1. Adiciona `and` / `or` booleanos com avaliação em curto-circuito (nas três).** O
`and` e o `or` *não* podem avaliar o lado direito quando o esquerdo já decide o
resultado (`false and X` é `false` sem tocar em `X`). Porque é que o curto-circuito é
o mesmo tipo de preguiça que o `if`, e onde pertence na escada de precedência?

<details>
<summary>Esboço da solução</summary>

O `and`/`or` são preguiçosos exatamente como o `if`: avalia o operando esquerdo, e só
avalia o direito se o esquerdo ainda não resolveu a resposta. Ficam *abaixo* da
comparação em precedência (para `a < b and c < d` agrupar as comparações primeiro), e
são implementados não como operadores binários comuns (que avaliam ansiosamente ambos
os lados) mas com o mesmo salto-de-ramo que o caso `if` usa. Essa preguiça partilhada
é porque `x != 0 and 10 / x > 1` é seguro — a divisão nunca corre quando `x` é 0.

</details>

**2. Torna a recursão mais barata com tail calls (pesquisa + uma linguagem).** O
`fact` como escrito constrói uma call stack profunda. Reescreve-o com um acumulador
(`fact(n, acc) = if n <= 1 then acc else fact(n - 1, n * acc)`) para que a chamada
recursiva seja a *última* coisa que faz. O que um interpretador poderia fazer com essa
forma que não consegue com o original?

<details>
<summary>Esboço da solução</summary>

Quando a chamada recursiva está em *posição de cauda* — a última operação, sem nada
para fazer com o seu resultado — o interpretador pode reutilizar o scope da chamada
atual em vez de aninhar um novo, transformando a recursão num loop que corre em espaço
de stack constante (tail-call optimization). A versão com acumulador tem essa forma; o
original `n * fact(n-1)` não, porque ainda tem de multiplicar depois de a chamada
voltar. Reconhecer tail calls é como as linguagens funcionais recorrem arbitrariamente
fundo sem transbordar a stack.

</details>

**3. Adiciona um loop estilo `while` e compara-o à recursão (qualquer linguagem).** Dá
à linguagem uma construção de loop, depois implementa `sum de 1 a n` com um loop e com
recursão. Com condicionais já presentes, um loop acrescenta algum *poder* à linguagem —
ou só conveniência?

<details>
<summary>Esboço da solução</summary>

Um loop não acrescenta poder computacional: a linguagem tornou-se Turing-completa no
momento em que teve recursão mais condicionais, e qualquer loop pode ser reescrito como
uma função recursiva (e vice-versa). O que um loop acrescenta é *conveniência e
eficiência* — uma iteração explícita fácil de ler e que corre em espaço de stack
constante sem precisar de tail-call optimization. É um ponto profundo: muitas features
de linguagem são expressividade e ergonomia em cima de um pequeno núcleo
Turing-completo, não capacidade nova. Já construíste esse núcleo.

</details>

## Canto da entrevista

<Interview question="Porque é que um condicional tem de avaliar só um ramo, e o que parte se avaliar ambos?">

O propósito todo de um condicional é a *escolha*, e escolha implica não fazer a coisa
que não escolheste. Concretamente, `if cond then a else b` tem de avaliar `cond`, depois
avaliar exatamente um de `a` ou `b` — nunca ambos. Duas coisas partem se avaliar ambos.
Primeiro, a correção: um ramo pode só ser *válido* quando é selecionado — `if n == 0
then 0 else 100 / n` divide por zero se o ramo else correr quando `n` é 0, então a
avaliação ansiosa transformaria um programa seguro num que rebenta. Segundo, e mais
fundamentalmente, a terminação: a recursão depende de o ramo recursivo *não* correr
quando o caso-base é atingido. `fact(n) = if n <= 1 then 1 else n * fact(n-1)` recorreria
para sempre se o interpretador avaliasse o ramo `else` mesmo quando `n <= 1`. Então a
preguiça do condicional não é uma otimização — é o que faz os condicionais significarem
seja o que for, e é a razão precisa pela qual adicionar `if` a uma linguagem com funções
desbloqueia recursão terminante e, com ela, a Turing-completude. É também por isto que o
`and`/`or` fazem curto-circuito e por que as linguagens distinguem formas especiais (como
o `if`) de funções comuns (cujos argumentos são todos avaliados): algumas construções
*têm* de controlar o que é avaliado.

</Interview>

## O track à frente

A tua linguagem é agora Turing-completa: um lexer, um parser recursive-descent, e um
evaluator tree-walking com variáveis, closures, e condicionais — tudo o que é preciso,
em princípio, para computar qualquer coisa, construído e testado de três formas,
byte-idêntico. O que resta é expressividade e ergonomia, não poder: booleanos em
curto-circuito, loops `while` e `for`, `print` e strings, estruturas de dados, melhores
mensagens de erro com números de linha. Cada um é uma adição modesta às mesmas quatro
fases que agora conheces intimamente. Propuseste-te desmistificar como uma linguagem
funciona — e construíste uma real, ainda que pequena, a partir de nada além de um scanner
de caracteres e destas ideias.
