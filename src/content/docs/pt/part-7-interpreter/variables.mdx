O [avaliador de expressões](/systems-programming-academy/pt/part-7-interpreter/expressions/)
computa `2 * (3 + 4)` e esquece-o instantaneamente. Uma *linguagem* lembra-se:
nomeias um valor e usa-lo mais tarde. Esta lição adiciona isso — variáveis,
atribuição, e a única estrutura que os faz funcionar, um **environment** — e ao
fazê-lo transforma uma calculadora numa pequena linguagem de programação:

```text
x = 5
y = x * 2 + 1     ← lê x, liga y
y - x             ← 6
```

<Aside type="note" title="O environment é a ideia toda">
  Tudo aqui assenta num conceito: um **environment**, um mapa `nome → valor`
  passado através da avaliação. A atribuição escreve nele; uma referência a
  variável lê dele. É só isto — mas este humilde mapa é a semente de *tudo* o que se
  segue no track. Adiciona funções e cada chamada ganha o seu próprio environment
  encadeado a um envolvente; essa cadeia é exatamente como **scopes**, **closures**,
  e a **call stack** funcionam. Constrói o environment agora e construíste o
  esqueleto da memória de um interpretador real.
</Aside>

## Três pequenas adições, uma por fase

Uma variável toca cada uma das três fases exatamente uma vez — o sinal limpo de um
design bem separado. O **lexer** aprende a ler identificadores e `=`; o **parser**
aprende que um nome pode ser uma expressão (uma *referência a variável*) e que
`name = expr` é uma *statement de atribuição*; o **evaluator** ganha um environment
para ler e escrever.

Ler um identificador é o mesmo loop "agrupar caracteres que pertencem juntos" que
ler um número multi-dígito — começa numa letra, continua enquanto o caractere for
uma letra, dígito, ou underscore.

### O parser: uma variável é uma expressão; `=` é uma statement

Uma referência a variável encaixa no `factor` mesmo ao lado de um número — ambos
são átomos. E aparece um tipo de nó totalmente novo: a **statement**. Um programa é
uma sequência de statements, cada uma ou uma atribuição ou uma expressão nua.

```rust title="src/lib.rs"
pub fn parse_stmt(tokens: Vec<Token>) -> Result<Stmt, String> {
    // `name = expr` é uma atribuição; qualquer outra coisa é uma expressão nua.
    let is_assign = matches!(tokens.first(), Some(Token::Ident(_)))
        && matches!(tokens.get(1), Some(Token::Assign));

    let stmt = if is_assign {

        p.pos += 1;                        // consome o '='
```

Distinguir atribuição de expressão com um look-ahead de dois tokens (`Ident` depois
`=`) é o parser de statements mais simples possível. Linguagens maiores generalizam
isto, mas a forma é a mesma: decide que *tipo* de statement isto é, depois parseia
as suas partes.

### O evaluator: um environment para ler e escrever

A avaliação agora carrega um environment. Uma referência a variável procura o seu
nome (erro se não estiver ligado); uma atribuição avalia o lado direito, guarda-o, e
devolve-o. E como as statements partilham um environment, **o estado persiste**.

```python title="vars.py"
def execute(stmt: Stmt, env: Env) -> int:
    if stmt.kind == "a":                               # atribuição
        v = evaluate(stmt.expr, env)
        env[stmt.name] = v                             # escreve no environment
        return v
    return evaluate(stmt.expr, env)
```

```rust title="src/lib.rs"
pub fn eval(e: &Expr, env: &Env) -> Result<i64, String> {
    match e {
        // referência a variável
        Expr::Var(name) => env.get(name).copied()
            .ok_or_else(|| format!("undefined variable '{name}'")),
        ...
    }
}

pub fn exec(stmt: &Stmt, env: &mut Env) -> Result<i64, String> {
    match stmt {
        Stmt::Assign(name, e) => {          // atribuição
            let v = eval(e, env)?;
            env.insert(name.clone(), v);    // escreve no environment
            Ok(v)
        }
        Stmt::Expr(e) => eval(e, env),
    }
}
```
</TabItem>

Repara nas formas de ownership: o Rust torna a mutação explícita — o `eval` recebe
`&Env` (só leitura), o `exec` recebe `&mut Env` (escreve), e o compilador força essa
divisão. O Go e o Python passam o mapa por referência e mutam-no livremente. Mesmo
comportamento; o Rust só torna "quem pode escrever" visível no tipo.

## O estado em ação

Corre um programa inteiro — uma statement por linha, partilhando um environment — e
a memória mostra-se:

```text
$ vars "x = 5; y = x * 2 + 1; y = y + 100; y - x"
x = 5  =>  5
y = x * 2 + 1  =>  11
y = y + 100  =>  111
y - x  =>  106
```

A terceira linha é a lição silenciosa: `y = y + 100` lê o `y` *antigo* (11), soma
100, e guarda o resultado de volta (111). A atribuição não é resolver equações — é
"avalia o lado direito no environment atual, depois religa o nome". Essa ordem
direita-depois-esquerda é porque `x = x + 1` faz sentido de todo, e é idêntica em
todas as linguagens aqui.

## Exercícios

**1. Adiciona atribuição composta `+=` (nas três).** Deixa `x += 3` significar
`x = x + 3`. Onde é o sítio mais limpo para o implementar — o lexer, o parser, ou
por *desugaring* no parser para a atribuição que já tens?

<details>
<summary>Esboço da solução</summary>

O movimento elegante é o *desugaring*: parseia `x += 3` e constrói a mesma árvore
exata que `x = x + 3` — uma atribuição cujo lado direito é `x + 3`. Então o
evaluator não precisa de mudanças nenhumas, porque o `+=` nunca foi um conceito
real para ele, só açúcar que o parser expandiu. É assim que muitas linguagens reais
implementam operadores compostos, e mostra uma ideia profunda: sintaxe nova é muitas
vezes mais barata de adicionar como uma transformação para sintaxe existente, não
como um caso novo no evaluator.

</details>

**2. Deteta uso não inicializado no parse (pesquisa + uma linguagem).** Agora
`z + 1` falha na *avaliação*. Conseguirias apanhar uma variável indefinida mais
cedo, durante o parse? Que informação o parser precisaria, e porque é que a maioria
das linguagens ainda adia isto para uma passagem posterior?

<details>
<summary>Esboço da solução</summary>

O parser sozinho não consegue saber em geral — uma variável pode ser definida numa
linha anterior que ele já parseou, então precisarias de seguir um conjunto de nomes
definidos à medida que avanças (uma pequena passagem de *análise semântica*). Casos
simples apanham-se facilmente, mas o problema geral (variáveis definidas noutros
scopes, definições condicionais) é porque compiladores reais correm uma passagem
separada de *resolução* ou *análise de nomes* entre o parse e a avaliação/codegen. É
um preview de porque o pipeline de três fases ganha mais fases à medida que uma
linguagem cresce.

</details>

**3. Adiciona uma statement `print` e block scope (qualquer linguagem).** Adiciona
`print expr` para um programa poder imprimir valores intermédios, depois adiciona
blocos `{ ... }` que ganham o seu *próprio* environment sobreposto ao exterior. O
que acontece a uma variável atribuída dentro de um bloco — e o que te diz isso sobre
como os scopes aninham?

<details>
<summary>Esboço da solução</summary>

O `print` é um novo tipo de statement que avalia a sua expressão e emite-a. O block
scope é a parte mais profunda: um bloco ganha um environment fresco cujo *parent* é
o envolvente, então um lookup que falha localmente cai para o parent — mas uma
atribuição dentro do bloco escreve na camada própria do bloco, então desaparece
quando o bloco acaba (a menos que atribuas deliberadamente a um nome exterior). Essa
cadeia de parents de environments é *scoping léxico*, e é exatamente a estrutura de
que funções e closures vão precisar — que é para onde o track vai a seguir.

</details>

## Canto da entrevista

<Interview question="Uma calculadora avalia expressões; uma linguagem tem variáveis. Arquiteturalmente, o que é preciso para as adicionar, e porque é uma mudança pequena?">

É preciso um **environment** — um mapa de runtime de nomes para valores — mais três
adições localizadas ao pipeline existente. O lexer aprende a tokenizar
identificadores e o operador de atribuição; o parser aprende que um identificador é
uma expressão atómica válida (uma referência a variável) e que um identificador
seguido de `=` começa uma statement de atribuição em vez de uma expressão; e o
evaluator passa o environment através, lendo-o numa referência a variável e
escrevendo-o numa atribuição. É uma mudança pequena precisamente porque o design de
três fases tem costuras limpas: cada fase ganha uma responsabilidade sem perturbar
as outras. A razão profunda de importar é que o environment generaliza: dá a cada
chamada de função o seu próprio environment ligado àquele onde foi definida, e tens
scoping léxico e closures; empilha esses environments e tens a call stack. Então
"adicionar variáveis" é arquiteturalmente o momento em que uma calculadora se torna
uma linguagem — um mapa é a diferença — e tudo o mais rico (funções, scopes,
objetos) é construído dando mais estrutura a esse mapa, não reconstruindo o
interpretador.

</Interview>

## O track à frente

Tens agora uma linguagem com memória: um lexer, um parser a produzir statements e
expressões, e um evaluator com um environment persistente — construído e testado de
três formas, byte-idêntico. O próximo passo é o grande: **funções**. Uma função é um
pedaço de árvore que podes nomear e chamar, e cada chamada precisa do seu *próprio*
environment encadeado àquele onde a função foi definida — que é de onde vêm as
closures e o scoping léxico, e onde este humilde mapa `nome → valor` revela todo o
seu poder. Daí: controlo de fluxo, e uma linguagem que é inconfundivelmente tua.
