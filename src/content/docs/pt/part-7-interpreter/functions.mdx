<Badge text="Constrói o Teu Próprio X" variant="tip" size="large" />

As [variáveis](/systems-programming-academy/pt/part-7-interpreter/variables/) deram
memória à linguagem. As **funções** dão-lhe *abstração* — nomeias uma computação e
reutiliza-la: `double(x) = x * 2`, depois `double(21)`. É o passo mais rico do
track, porque as funções forçam o environment da lição 2 a crescer numa **cadeia**,
e essa cadeia é o mecanismo por trás das duas ideias mais mal-entendidas da
programação: **closures** e **scoping léxico**. Construímos ambas, byte-idênticas,
de três formas.

<Aside type="note" title="Duas ideias, ambas sobre environments">
  Uma função precisa de duas coisas além de uma expressão. Primeiro, um **scope
  fresco por chamada** para guardar os seus argumentos, para que os parâmetros não
  vazem e chamadas recursivas ou aninhadas não se atropelem. Segundo — a subtil —
  uma função é uma **closure**: captura o environment onde foi *definida*, e o seu
  scope de chamada encadeia-se a *esse*, não a onde quer que tenha sido chamada.
  Então uma função vê as variáveis em scope onde foi escrita. Isso é **scoping
  léxico**, e é a única decisão de design que separa linguagens previsíveis de
  confusas.
</Aside>

## Os valores tornam-se first-class

Até agora todo o valor era um inteiro. Agora um valor é *ou um inteiro ou uma
função* — e uma função pode ser guardada numa variável, passada como argumento, e
devolvida. É isso que "first-class" significa. Então o tipo de valor do evaluator
ganha uma variante:

O campo-chave é o `env` — a função *lembra-se do environment onde nasceu*. Essa
única referência é a closure. Tudo o resto decorre dela.

## O environment torna-se uma cadeia

Um único mapa já não chega: uma chamada de função precisa do seu próprio scope para
os parâmetros, sobreposto ao scope que capturou. Então o `Scope` ganha um
**parent**, e o lookup percorre a cadeia para fora até encontrar o nome.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="fns.py"
class Scope:
    def __init__(self, parent: Scope | None = None) -> None:
        self.vars: dict[str, int | Function] = {}
        self.parent = parent

    def lookup(self, name: str) -> int | Function | None:
        scope = self
        while scope is not None:
            if name in scope.vars:
                return scope.vars[name]
            scope = scope.parent          # falha local? percorre para fora
        return None
```
</TabItem>
<TabItem label="Go">
```go title="fns.go"
type Scope struct {
	vars   map[string]Value
	parent *Scope
}

func lookup(env *Scope, name string) (Value, bool) {
	for s := env; s != nil; s = s.parent { // percorre para fora
		if v, ok := s.vars[name]; ok {
			return v, true
		}
	}
	return Value{}, false
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
pub struct Scope {
    vars: HashMap<String, Value>,
    parent: Option<Env>,
}
pub type Env = Rc<RefCell<Scope>>;

fn lookup(env: &Env, name: &str) -> Option<Value> {
    let scope = env.borrow();
    if let Some(v) = scope.vars.get(name) {
        return Some(v.clone());
    }
    match &scope.parent {
        Some(p) => lookup(p, name),        // percorre para fora
        None => None,
    }
}
```
</TabItem>
</Tabs>

<Aside type="tip" title="Porque o Rust precisa de Rc<RefCell<Scope>>">
  Uma closure e o código que a definiu *partilham* um environment, e ele pode ser
  mutado depois da captura. O Go e o Python exprimem estado mutável partilhado com
  um simples ponteiro / referência de objeto. O Rust torna-o explícito: o `Rc` dá
  ownership partilhado (muitos donos do mesmo scope), e o `RefCell` dá interior
  mutability (mutação através de um handle partilhado, verificada em runtime). É
  mais cerimónia, mas nomeia exatamente o que está a acontecer — *partilhado,
  mutável* — que as outras duas deixam implícito. O comportamento é idêntico; o
  Rust só se recusa a deixá-lo invisível.
</Aside>

## A chamada: um scope fresco encadeado à *definição*

Aqui está a lição toda numa função. Avaliar uma chamada: procura a função, avalia
os argumentos *no environment do chamador*, depois cria um novo scope para os
parâmetros cujo parent é o environment **capturado** da função — e corre o corpo aí.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="fns.py"
def _eval_call(e: Expr, env: Scope) -> int | Function:
    fn = env.lookup(e.name)
    if not isinstance(fn, Function):
        raise CalcError(f"'{e.name}' is not a function")   # (ou undefined)
    if len(e.args) != len(fn.params):
        raise CalcError(f"'{e.name}' expects {len(fn.params)} argument(s), ...")
    argv = [evaluate(a, env) for a in e.args]      # args no env do CHAMADOR
    call = Scope(parent=fn.env)                    # ...mas encadeia ao env de DEFINIÇÃO
    for p, v in zip(fn.params, argv):
        call.vars[p] = v
    return evaluate(fn.body, call)
```
</TabItem>
<TabItem label="Go">
```go title="fns.go"
func evalCall(e *Expr, env *Scope) (Value, error) {
	v, _ := lookup(env, e.Name)
	if v.Kind != 'f' { /* undefined / not a function */ }
	fn := v.Fn
	if len(e.Args) != len(fn.Params) { /* arity error */ }
	argv := make([]Value, len(e.Args))
	for i, a := range e.Args {
		argv[i], _ = Eval(a, env)              // args no env do CHAMADOR
	}
	call := childEnv(fn.Env)                    // ...mas encadeia ao env de DEFINIÇÃO
	for i, p := range fn.Params {
		call.vars[p] = argv[i]
	}
	return Eval(fn.Body, call)
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
Expr::Call(name, args) => {
    let func = match lookup(env, name) { /* Func / not-a-fn / undefined */ };
    if args.len() != func.params.len() { /* arity error */ }
    let mut argv = Vec::new();
    for a in args {
        argv.push(eval(a, env)?);              // args no env do CHAMADOR
    }
    let call_scope = child_env(&func.env);      // ...mas encadeia ao env de DEFINIÇÃO
    for (p, v) in func.params.iter().zip(argv) {
        call_scope.borrow_mut().vars.insert(p.clone(), v);
    }
    eval(&func.body, &call_scope)
}
```
</TabItem>
</Tabs>

**Essa única escolha — o parent é o `fn.env`, não o `env` do chamador — é o scoping
léxico.** Erra-a (encadeia ao chamador) e tens scoping *dinâmico*, onde o
comportamento de uma função depende de quem a chama — uma fonte notória de bugs. O
teste prova que a acertámos:

```text
x = 10
f(n) = n + x        # f captura x = 10 de onde é DEFINIDA
g(x) = f(0)         # g tem o seu PRÓPRIO parâmetro também chamado x
g(999)              # => 10, NÃO 999
```

Quando `g(999)` corre, o scope de `g` liga `x = 999` e chama `f(0)`. Mas o corpo de
`f` corre num scope encadeado ao environment de definição de *`f`*, onde `x` é 10 —
o `x` de `g` é invisível para ele. Então a resposta é `10`. É essa a diferença entre
scoping léxico e dinâmico, e é decidida inteiramente por qual environment o parent
do scope de chamada aponta.

## As closures capturam um environment *vivo*

Como o environment capturado é partilhado (não copiado), uma closure vê mudanças
posteriores nas variáveis que fechou:

```text
base = 100
shift(n) = n + base   # captura o environment que contém base
shift(5)              # => 105
base = 200            # muta o environment capturado
shift(5)              # => 205  — a closure vê o novo valor
```

Isto sai de graça nas três linguagens precisamente porque o environment é uma coisa
partilhada e mutável — o `Rc<RefCell>` do Rust, o ponteiro do Go, a referência do
Python apontam todos para o *mesmo* objeto scope, então reatribuir `base` é visível
através da closure. Mesmo comportamento, output byte-idêntico, três grafias
diferentes de "partilhado e mutável".

## Exercícios

**1. Adiciona recursão — e bate no problema do caso-base (nas três).** Tenta
`fact(n) = n * fact(n - 1)`. Já *faz parse* e o nome está em scope (as funções
capturam o environment onde são definidas, que agora contém a própria). Mas
`fact(3)` nunca pára. Adiciona um condicional para a recursão poder terminar — que
construção nova mínima a linguagem precisa?

<details>
<summary>Esboço da solução</summary>

A recursão funciona estruturalmente no momento em que uma função consegue ver o seu
próprio nome (consegue, porque a closure captura o scope global onde a função está a
ser definida). O que falta é uma forma de *parar*: uma expressão `if`/condicional,
para `fact(n) = if n <= 1 then 1 else n * fact(n - 1)`. Isso precisa de um operador
de comparação e um nó condicional de três ramos na AST mais um caso de `eval` que
avalia só o ramo tomado (lazy, não ambos). Os condicionais são a próxima primitiva —
com eles mais funções, a linguagem torna-se Turing-completa.

</details>

**2. Torna as funções verdadeiramente first-class: passa uma como argumento
(pesquisa + uma linguagem).** Como um `Value` pode ser uma função, `apply(f, x) = f(x)`
devia funcionar — chamar a função passada. O teu evaluator já suporta isto, ou a
maquinaria de chamada assume que o callee é procurado por nome? O que tem de mudar?

<details>
<summary>Esboço da solução</summary>

O bloqueio é geralmente que o `Call` guarda um *nome* (uma string) e procura-o, em
vez de avaliar uma expressão arbitrária a um valor-função. Generaliza o nó de
chamada para que o callee seja uma *expressão* que avalia a uma função — então
`f(x)` onde `f` é um parâmetro a segurar uma função simplesmente funciona, e devolver
uma função para chamar depois também. Essa única mudança (callee é um valor, não um
nome) é o que torna as funções totalmente first-class e desbloqueia funções de ordem
superior como `map` e `compose` — a recompensa de ter feito das funções valores em
primeiro lugar.

</details>

**3. Adiciona closures que capturam estado por-chamada (qualquer linguagem).** Faz
uma função que *devolve* uma função a capturar o seu argumento: `make_adder(k)`
devolve uma função que soma `k`. Duas chamadas, `make_adder(10)` e `make_adder(100)`,
têm de produzir dois adders independentes. Porque é que cada `k` capturado fica
separado?

<details>
<summary>Esboço da solução</summary>

Cada *chamada* a `make_adder` cria um scope fresco a ligar o seu próprio `k`, e a
função interna devolvida captura *esse* scope. Como as duas chamadas criaram dois
objetos scope diferentes, o adder de `make_adder(10)` fecha sobre um scope onde
`k=10` e o de `make_adder(100)` sobre um scope diferente onde `k=100` — nunca
interferem. É a recompensa mais profunda do design de cadeia-de-environments: as
closures dão a cada instância de função o seu próprio estado privado e persistente,
que é a fundação de contadores, iteradores, e encapsulamento tipo-objeto construído
puramente a partir de funções.

</details>

## Canto da entrevista

<Interview question="Explica closures e scoping léxico, usando como as implementarias.">

Uma closure é uma função agrupada com o environment em que foi definida. Em termos
de implementação: representa um environment como um scope com um mapa de ligações e
um ponteiro para um scope parent, para que o lookup de nomes percorra a cadeia para
fora. Quando *avalias uma definição de função*, guarda, ao lado dos parâmetros e do
corpo, uma referência ao environment atual — essa referência capturada é a closure.
Quando *chamas* a função, cria um scope fresco para os parâmetros e define o seu
parent como o environment capturado da função (não o environment do chamador),
depois avalia o corpo aí. Essa única decisão — o parent do scope de chamada é o
environment de *definição* — é o scoping léxico: a função resolve variáveis livres
onde foi escrita, não onde foi chamada, o que é previsível e é o que toda a
linguagem mainstream faz. (Encadear ao chamador dá scoping dinâmico.) Como o
environment capturado é segurado por referência e é mutável, a closure também vê
mudanças posteriores nessas variáveis, e cada chamada a uma função envolvente cria
um scope distinto, então closures devolvidas ganham estado privado independente.
Esse mecanismo de cadeia-de-environments — scopes com ponteiros parent, capturados
na definição, estendidos na chamada — é tudo, e é idêntico quer grafes "environment
mutável partilhado" como um ponteiro Go, uma referência Python, ou o `Rc<RefCell>`
do Rust.

</Interview>

## O track à frente

A tua linguagem tem agora funções e closures — o conceito mais difícil do track, e o
que revela porque o environment teve de se tornar uma cadeia. O que resta é
sobretudo composição do que construíste: **condicionais** (que desbloqueiam a
recursão e tornam a linguagem Turing-completa), **operadores de comparação e
booleanos**, valores-função first-class passados e devolvidos (funções de ordem
superior), e depois statements como loops e `print`. Cada um é uma pequena adição às
mesmas três fases. Construíste o coração pulsante de um interpretador; o resto é
dar-lhe mais para dizer.
