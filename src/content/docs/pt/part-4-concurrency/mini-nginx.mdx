## Três modelos de concorrência, um trabalho

<Mermaid title="O mesmo caminho de pedido, tres formas de o correr em concorrencia" code={`flowchart TD
  A["accept() de uma ligacao"] --> B{"modelo de concorrencia"}
  B -->|Rust| C["entrega a um thread pool fixo por um channel"]
  B -->|Go| D["go handleConnection(conn) - uma goroutine"]
  B -->|Python| E["agenda uma corrotina no event loop"]
  C --> F["parse da request line + headers"]
  D --> F
  E --> F
  F --> G["resolve o path dentro do docroot (rejeita '..')"]
  G --> H["le o ficheiro, monta a resposta HTTP"]
  H --> I["escreve a resposta, fecha a ligacao"]`} />

O tratamento do pedido é idêntico nos três; só a **linha que consegue a
concorrência** difere. É essa a experiência toda.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="mininginx.py — corrotina por ligação"
async def start_server(docroot: Path, port: int = 0) -> tuple[asyncio.Server, int]:
    """Arranca o servidor; devolve-o e o porto realmente bound."""

    async def handler(reader, writer) -> None:
        await handle_connection(reader, writer, docroot)

    server = await asyncio.start_server(handler, "127.0.0.1", port)
    bound_port = server.sockets[0].getsockname()[1]
    return server, bound_port
```

O `asyncio.start_server` agenda uma corrotina por ligação num event loop de
uma só thread (lição 3). Cada `await` dentro do handler é um ponto de suspensão
onde outras ligações correm. Uma thread, sem locks precisos.
</TabItem>
<TabItem label="Go">
```go title="main.go — goroutine por ligação"
// serve aceita ligações para sempre, uma goroutine cada.
func serve(ln net.Listener, docroot string) {
	for {
		conn, err := ln.Accept()
		if err != nil {
			return
		}
		go handleConnection(conn, docroot)
	}
}
```

Uma linha — `go handleConnection(conn, docroot)`. O netpoller (lição 3)
transforma as leituras bloqueantes de cada goroutine em esperas epoll; o
runtime agenda milhares delas pelos cores. A concorrência é **ilimitada** por
omissão, que é a postura idiomática do Go e um trade-off real que vamos medir.
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs — thread pool fixo"
/// Corre o servidor: `n_workers` threads puxam ligações de um channel e
/// servem-nas. É a forma do projeto final do Rust Book, sobre as lições
/// desta Parte.
pub fn serve(listener: TcpListener, docroot: PathBuf, n_workers: usize) {
    assert!(n_workers > 0, "need at least one worker");
    let (tx, rx) = mpsc::channel::<TcpStream>();
    let rx = Arc::new(Mutex::new(rx));
    let docroot = Arc::new(docroot);

    for _ in 0..n_workers {
        let rx = Arc::clone(&rx);
        let docroot = Arc::clone(&docroot);
        thread::spawn(move || {
            // recv() devolve Err quando o sender é dropped: shutdown limpo.
            while let Ok(stream) = { rx.lock().unwrap().recv() } {
                handle_connection(stream, &docroot);
            }
        });
    }

    for stream in listener.incoming().flatten() {
        if tx.send(stream).is_err() {
            break;
        }
    }
}
```

O channel (lição 2) distribui o trabalho; o `Arc<Mutex<Receiver>>` (lição 1)
deixa N workers partilharem o único consumidor mpsc. A concorrência é
**limitada** por `n_workers` — um dial de capacidade deliberado.
</TabItem>
</Tabs>

## O handler de pedido partilhado

Tudo abaixo da linha de concorrência é a mesma lógica, portada fielmente. A
parte interessante é a **segurança**: mapear um URL para um ficheiro é onde os
servidores estáticos são comprometidos. O nosso `resolve` é puramente lexical
e rejeita qualquer componente `..` *antes de tocar no sistema de ficheiros* —
o pedido nunca tem hipótese de escapar do document root.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="mininginx.py"
def resolve(docroot: Path, url_path: str) -> Path | None:
    """Mapeia um path de URL para um ficheiro dentro do ``docroot``, ou
    ``None`` se tentar escapar. Puramente lexical: qualquer ``..`` é rejeitado.

    >>> resolve(Path("/srv/www"), "/") == Path("/srv/www/index.html")
    True
    >>> resolve(Path("/srv/www"), "/../etc/passwd") is None
    True
    """
    if url_path == "/":
        url_path = "/index.html"
    resolved = docroot
    for component in url_path.split("/"):
        if component in ("", "."):
            continue
        if component == ".." or "\0" in component:
            return None  # tentativa de traversal: nunca sai do docroot
        resolved = resolved / component
    return resolved
```
</TabItem>
<TabItem label="Go">
```go title="main.go"
// resolve mapeia um path de URL para um ficheiro dentro do docroot, ou
// devolve ok=false se tentar escapar. Puramente lexical: `..` é rejeitado.
func resolve(docroot, urlPath string) (string, bool) {
	if urlPath == "/" {
		urlPath = "/index.html"
	}
	resolved := docroot
	for _, component := range strings.Split(urlPath, "/") {
		switch {
		case component == "" || component == ".":
			continue
		case component == "..":
			return "", false // tentativa de traversal: nunca sai do docroot
		case strings.ContainsRune(component, 0):
			return "", false
		default:
			resolved = filepath.Join(resolved, component)
		}
	}
	return resolved, true
}
```
</TabItem>
<TabItem label="Rust">
```rust title="src/lib.rs"
/// Mapeia um path de URL para um ficheiro dentro do `docroot`, ou `None` se
/// tentar escapar. Puramente lexical: qualquer `..` é rejeitado logo.
pub fn resolve(docroot: &Path, url_path: &str) -> Option<PathBuf> {
    let path = if url_path == "/" { "/index.html" } else { url_path };
    let mut resolved = docroot.to_path_buf();
    for component in path.split('/') {
        match component {
            "" | "." => continue,
            ".." => return None, // tentativa de traversal: nunca sai do docroot
            c if c.contains('\0') => return None,
            c => resolved.push(c),
        }
    }
    Some(resolved)
}
```
</TabItem>
</Tabs>

Cada servidor tem um teste que dispara `GET /../secret.txt` contra um
`secret.txt` colocado *fora* do document root e verifica um `404` sem bytes
vazados. Isto não é decoração: directory traversal (CWE-22) é um dos bugs web
mais antigos e mais explorados, e "rejeitar `..` lexicalmente antes de
qualquer chamada ao sistema de ficheiros" é a defesa que de facto aguenta.

<Aside type="tip" title="Porquê lexical e não canonicalização?">
  Uma alternativa tentadora é canonicalizar o path resolvido e verificar que
  ainda começa pelo docroot. Também funciona, mas toca no sistema de ficheiros
  (segue symlinks, que introduzem a sua própria fuga por um subdiretório com
  symlink) e é fácil de errar subtilmente. Rejeitar `..` antes de resolver é
  mais simples de auditar e não tem dependência do sistema de ficheiros —
  menos peças móveis em código crítico de segurança é, por si, a
  funcionalidade.
</Aside>

## Ler os resultados com honestidade

**O Go ganhou em débito, e o modelo dele é a razão.** Goroutine-por-ligação
com o netpoller é exatamente a carga para que o runtime do Go foi construído:
criação barata, multiplexing epoll, work-stealing pelos 20 cores. Aguentou
concorrência 200 quase tão bem como 50 — o runtime absorveu a carga extra.

**O Rust usou uma fração da memória** (2,3 MB vs 10,5 MB vs 22,2 MB) — sem
runtime, sem GC, sem stacks por goroutine; só 8 threads e o SO. Para um
sidecar, um proxy de edge, ou qualquer coisa limitada em memória, essa
diferença de 4–10× é muitas vezes o número decisivo, não o pico de req/s.

**O Python foi 4–5× mais lento e degradou sob carga** (13k → 7,7k à medida que
a concorrência subiu), porque um event loop de uma thread serializa *todo* o
tratamento de pedidos num core — 19 cores ficaram parados. O asyncio brilha
quando os handlers são I/O-bound (à espera de uma base de dados, de uma API a
montante); servir um ficheiro minúsculo em cache é quase puro CPU por pedido,
o pior caso do asyncio. A correção em produção é correr N processos de event
loop atrás de um load balancer — que é exatamente a forma dos deployments web
de Python reais (gunicorn + workers uvicorn).

**O número mais surpreendente é o próprio sweep de thread-pool do Rust** — o
débito *caiu* à medida que os workers subiram de 1:

Estas execuções são ruidosas (a ordem de 8/20/50 baralha-se entre runs), mas
o título é estável e é a lição 1 a voltar em força: **quando o trabalho por
pedido é minúsculo, o `Mutex<Receiver>` partilhado é o gargalo.** Cada worker
tem de trancar o mesmo mutex para puxar a próxima ligação, por isso acrescentar
workers acrescenta *contenção* nesse lock mais depressa do que acrescenta
paralelismo útil. Um worker nunca contende — só serve. No momento em que os
handlers fazem trabalho a sério (ler um ficheiro grande, fazer proxy a
montante), o equilíbrio inverte-se e mais workers ganham. O dial tem uma
posição certa, e depende da carga — precisamente por isso o nginx faz do
`worker_processes` um afinável e não uma constante.

<Aside type="note" title="O que isto NÃO prova">
  Estes números são uma carga (ficheiro estático minúsculo, loopback, uma
  máquina) numa tarde. São honestos para o que medem e inúteis como
  classificação geral. "Rust é mais lento que Go a servir HTTP" seria a lição
  errada; "a arquitetura de concorrência tem de casar com a forma da carga, e
  tens de medir a tua" é a certa.
</Aside>

## Exercícios

**1. Dá ao Rust uma thread por ligação (Rust).** Substitui o thread pool por
`thread::spawn` por ligação aceite (ilimitado, como o Go). Volta a correr o
load test. Bate o pool? A que concorrência começa o custo de criação a
aparecer? O que parte com 10.000 ligações simultâneas?

<details>
<summary>Esboço de solução</summary>

Threads por ligação muitas vezes *batem* o pool contendido de 8 workers nesta
carga minúscula (sem lock de receiver partilhado), confirmando o diagnóstico
de contenção. Mas cada thread reserva ~8 MB de stack virtual e um slot no
scheduler do kernel, por isso a milhares de ligações lentas simultâneas
colapsa onde as goroutines (uns KB cada) continuam — a razão exata pela qual o
modelo do Go escala e a motivação C10K da lição 3.

</details>

**2. Acrescenta keep-alive (nas três).** Suporta `Connection: keep-alive` do
HTTP/1.1: itera na mesma ligação, lendo e respondendo a vários pedidos antes
de fechar. Como muda isto a matemática da concorrência — e que modelo
beneficia mais?

<details>
<summary>Esboço de solução</summary>

O keep-alive amortiza o custo de accept/spawn por muitos pedidos, por isso
ajuda todos os modelos, mas mais aqueles onde o setup por ligação era caro
(thread-por-ligação). Também aumenta o risco no limite de concorrência: uma
ligação keep-alive *segura* um worker/goroutine enquanto está inativa, por
isso o pool de Rust pode ser esfomeado por N clientes keep-alive inativos —
introduzindo a necessidade de timeouts por ligação, que é o teu próximo
exercício.

</details>

**3. Serve esta própria academia (nas três).** Aponta o docroot de cada
servidor para a pasta `dist/` do Astro e navega o site real pelo teu
mini-NGINX. O que parte? (Pista: MIME types que não tratas, e resolução de
index para URLs de diretório como `/part-4-concurrency/`.) Corrige o menor
conjunto de coisas para a homepage renderizar.

<details>
<summary>Esboço de solução</summary>

Vais bater em content types em falta (`.svg`, `.woff2`, `.xml`) servidos como
`application/octet-stream` — o browser recusa aplicar CSS/fontes — e URLs de
diretório que precisam de mapear para `index.html` dentro do diretório, não só
na raiz. Acrescentar um punhado de entradas MIME e uma regra "se o path é um
diretório, acrescenta index.html" põe a página a renderizar. É o momento em
que o brinquedo se torna reconhecivelmente um servidor estático a sério.

</details>

## Cantinho da entrevista

<Interview question="Explica-me como defender um servidor de ficheiros estáticos contra directory traversal.">

O ataque é um URL como `/../../etc/passwd` a tentar ler fora do document root
(CWE-22). A defesa em que confio é lexical e acontece antes de qualquer acesso
ao sistema de ficheiros: divide o path por `/`, e rejeita o pedido inteiro se
algum componente for `..` (rejeita também bytes nulos, que podem truncar paths
nalgumas syscalls). Como nunca resolve o path malicioso, não há comportamento
de sistema de ficheiros para errar. A alternativa tentadora — canonicalizar e
verificar o prefixo — também funciona mas segue symlinks, o que reintroduz a
fuga por um subdiretório com symlink, e verificações de prefixo têm bugs
clássicos (`/srv/www-evil` começa por `/srv/www`). Seja qual for a escolha,
tem de ser testada com um pedido de fuga real contra um ficheiro colocado fora
da raiz — que todas as versões deste servidor fazem.

</Interview>

<Interview question="Thread pool, goroutine-por-ligação, ou event loop — como escolhes para um servidor?">

Pela forma da carga e pelas restrições, e mediria, não adivinharia. Event
loop (asyncio, Node) serve handlers I/O-bound com contagens altas de ligações
— milhares de sockets quase sempre à espera — em poucos cores, mas serializa
trabalho de CPU e precisa de escala multi-processo para usar muitos cores.
Goroutine-por-ligação (Go) é o generalista: concorrência barata, multi-core
por omissão, estilo bloqueante; custou mais memória que o Rust no nosso teste
mas ganhou o débito. Thread-pool (clássico, o nosso Rust) dá uso de recursos
limitado — o dial de capacidade importa quando sistemas a jusante podem ser
sobrecarregados — e memória mínima, ao custo de um ponto de dispatch
partilhado que pode contender, como medimos. A resposta honesta numa
entrevista é "depende de os pedidos serem I/O- ou CPU-bound, da contagem de
ligações, do orçamento de memória e do número de cores — eis como o
benchmarkaria", e depois citar exatamente o tipo de números que este projeto
produziu.

</Interview>

<Interview question="O nosso servidor Rust ficou mais lento à medida que acrescentámos threads. Como diagnosticarias e corrigirias isso em produção?">

Esse sintoma — escala negativa com workers acrescentados — quase sempre
significa contenção num recurso partilhado, e confirmaria com um profiler
(`perf`, ou um profile de contenção de lock) a mostrar tempo passado à espera
de um mutex. No nosso caso é o único `Mutex<Receiver>` que cada worker tranca
para desenfileirar uma ligação; com trabalho trivial por pedido, o lock é o
caminho quente. Correções, por ordem: reduzir o tempo na secção crítica (já
mínimo aqui); fazer sharding da fonte de trabalho para os workers não
partilharem um lock (várias filas, ou um channel MPMC lock-free como o
crossbeam); ou, se o trabalho por pedido é genuinamente minúsculo, questionar
se precisas de um pool sequer versus accept-and-spawn. O meta-ponto: mais
paralelismo não é grátis, e um ponto de coordenação partilhado pode
transformar workers acrescentados em contenção acrescentada — mede a curva de
escala, não assumas que é monótona.

</Interview>

(1 KB, 100 KB, 1 MB), e traça débito e percentis de latência (p50/p99) para
cada servidor de concorrência 1 → 1000. Depois acrescenta uma funcionalidade
de produção que muda o quadro — um timeout por ligação, uma pequena cache de
respostas, ou gzip — e mede o custo. Vais sair com uma intuição de percentis
de latência que nenhuma quantidade de leitura dá, e um servidor que
compreendes genuinamente do `accept()` para cima. É esse o propósito de toda a
Parte 4: a concorrência deixa de ser vocabulário e torna-se um conjunto de
escolhas de engenharia medidas e deliberadas.
