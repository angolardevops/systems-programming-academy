Esta academia ensinou três linguagens não em paralelo mas **umas contra as
outras** — o mesmo programa, construído de três formas, medido na mesma máquina.
Essa estrutura foi a pedagogia inteira: um trade-off que consegues *ver num
número* é compreendido de uma forma que um parágrafo nunca entrega. Esta reflexão
final reúne esses números num só sítio e faz a pergunta que todo o engenheiro
acaba por enfrentar: **que linguagem, e porquê?**

Todos os valores abaixo foram medidos numa lição anterior, na mesma máquina Linux
de 20 cores, e estão ligados à sua origem. Nada aqui é sensação; são os dados da
própria academia, sintetizados.

<Aside type="caution" title="Os números são verdadeiros, os rankings não">
  Estas medições são honestas para o que mediram — cargas específicas, uma
  máquina, uma tarde. São inúteis como ranking universal. "Rust é o mais rápido" é
  a lição errada; "eis exatamente o que cada escolha custa neste tipo de trabalho,
  por isso mede o teu" é a certa. Lê os números como prova de *como pensar*, não
  como uma tabela classificativa.
</Aside>

## Arranque e footprint: o que pagas para existir

Antes de fazer trabalho nenhum, um programa paga para arrancar e para ocupar
memória. Para uma CLI invocada milhares de vezes num pipeline, ou um sidecar que
corres às centenas, este é muitas vezes o número que decide.

A forma repete-se em todo o lado onde a academia mediu memória (o
[exporter](/systems-programming-academy/pt/part-3-devops/prometheus-exporter/)
registou 2.0 / 7.6 / 21.3 MB para a mesma carga). O Rust não carrega runtime nem
garbage collector, por isso o seu chão é o SO e umas threads. O runtime e o GC do
Go custam um punhado de megabytes mas compram conveniência enorme. O interpretador
do Python são dezenas de megabytes antes de o teu código correr. Nada disto importa
para um serviço de longa duração onde o arranque amortiza a nada — e tudo isto
decide um cold start de function-as-a-service ou um container limitado em memória.

## Débito: o custo do mesmo trabalho

O [analisador de logs](/systems-programming-academy/pt/part-3-devops/log-analyzer/)
fez parse de 500.000 linhas de trabalho idêntico, verificado byte-idêntico nas
três:

<Benchmark
  metric="Parse de 500k linhas de log (menor é melhor)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust (release)", value: 98, unit: "ms" },
    { label: "Go (compilado)", value: 131, unit: "ms" },
    { label: "Python 3.12 (CPython)", value: 736, unit: "ms" },
  ]}
/>

Rust e Go ficam a ~30% um do outro; o Python é ~7× mais lento neste trabalho de
texto CPU-bound. Esse rácio — linguagens compiladas agrupadas, o CPython cerca de
uma ordem de grandeza atrás no cálculo cru — aguentou em todo o lado onde a
academia mediu computação pura. É também exatamente por isso que o ecossistema do
Python empurra os hot loops para C (NumPy, o motor `re`, extensões nativas): Python
idiomático chama código nativo rápido em vez de iterar em bytecode.

## Concorrência: onde os runtimes mais divergem

A [lição de threads](/systems-programming-academy/pt/part-4-concurrency/threads-shared-state/)
martelou um contador partilhado — quatro threads, um milhão de incrementos cada — e
os números contaram três histórias diferentes:

<Benchmark
  metric="4 threads × 1M incrementos de um contador partilhado (menor é melhor)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust — mutex por incremento", value: 356, unit: "ms" },
    { label: "Go — mutex por incremento", value: 112, unit: "ms" },
    { label: "Go — atómico", value: 49, unit: "ms" },
    { label: "Rust — atómico", value: 51, unit: "ms" },
    { label: "batched (ambos, lock uma vez)", value: 0.3, unit: "ms" },
  ]}
/>

Duas descobertas moldaram o arco de concorrência inteiro. **Os atómicos custam o
mesmo em Rust e Go (~50 ms)** porque ambos compilam para a mesma instrução de CPU —
a esse nível a linguagem é irrelevante, estás a medir coerência de cache. Mas **o
mutex contendido do Go bateu o do Rust por 3×**, porque o lock do Go está integrado
com o seu scheduler em userspace enquanto o `std::sync::Mutex` do Rust faz chamadas
futex ao kernel sob contenção brutal. O mesmo nome de primitiva, maquinaria
diferente por baixo — a lição recorrente da Parte 4.

E o Python? O GIL significa que quatro threads CPU-bound correm *mais lento* do que
uma ([medido](/systems-programming-academy/pt/part-4-concurrency/threads-shared-state/):
60 ms vs 54 ms). Threads em Python paralelizam I/O, nunca CPU — o que não é um bug a
corrigir mas uma arquitetura à volta da qual desenhar (multiprocessing, ou código
nativo que liberta o GIL).

O [servidor mini-NGINX](/systems-programming-academy/pt/part-4-concurrency/mini-nginx/)
juntou tudo sob carga HTTP real:

<Benchmark
  metric="20k pedidos HTTP, concorrência 50 (maior é melhor)"
  lowerIsBetter={false}
  rows={[
    { label: "Go (goroutine por ligação)", value: 71143, unit: "req/s" },
    { label: "Rust (thread pool)", value: 53373, unit: "req/s" },
    { label: "Python (asyncio, 1 core)", value: 12948, unit: "req/s" },
  ]}
/>

O modelo goroutine-por-ligação do Go — concorrência barata, multicore por omissão —
é feito de propósito para servidores e ganhou o débito. O Rust usou uma **fração da
memória** (2.3 vs 10.5 MB) mas o seu thread pool de lock partilhado contendeu. O
event loop de um core do Python serializou tudo, o preço do GIL para trabalho
CPU-ish. Três modelos, três formas, cada um melhor nalgum sítio.

## O que o compilador compra, numa imagem

Para além dos números, a academia mostrou um eixo qualitativo: **quando é que
descobres que estavas errado?**

<Mermaid title="Quando cada linguagem apanha o bug classico de concorrencia" code={`flowchart TD
  BUG["data race: duas threads mutam estado partilhado sem sincronizacao"]
  BUG --> RUST["Rust: erro de compilacao (E0499) - o bug nunca existe"]
  BUG --> GO["Go: compila & corre errado; race detector encontra SE um teste bater nele"]
  BUG --> PY["Python: o GIL estreita a janela; ainda um bug latente, sem detetor"]`} />

A [lição de threads](/systems-programming-academy/pt/part-4-concurrency/threads-shared-state/)
demonstrou isto concretamente: o mesmo contador com race é um **erro de compilação**
em Rust, perde silenciosamente 65% dos seus incrementos em Go (apanhado só pelo `go
test -race`), e é um bug latente silencioso em Python. Este é o trade-off mais fundo
dos três: o Rust move a falha para tempo de compilação ao custo de lutar com o borrow
checker; o Go e o Python adiam-na para runtime por escrita mais rápida. Nenhum é
grátis; ambos estão certos para riscos diferentes.

## Então: que linguagem?

A resposta honesta que a academia ganha é um conjunto de **condições**, não um
vencedor.

<Mermaid title="Um esboco de decisao, ancorado nas medicoes" code={`flowchart TD
  Q["o que domina o teu problema?"]
  Q -->|"velocidade de iteracao, glue, dados/ML, scripts"| PY["Python: velocidade de escrita imbativel; empurra hot loops p/ nativo"]
  Q -->|"servidores, concorrencia, equipas, deploys simples"| GO["Go: concorrencia barata, compilacao rapida, um binario, facil de ler"]
  Q -->|"limites de memoria/latencia, safety-critical, sistemas"| RUST["Rust: sem GC, sem runtime, races impossiveis - a custo de compilacao"]`} />

- O **Python** ganha quando *o teu* tempo é o gargalo, não o do CPU: exploração,
  dados e ML, automação, glue. A sua penalização de 7× no cálculo é irrelevante
  quando o hot path é código nativo ou o programa corre uma vez. É a linguagem mais
  rápida de *escrever*, que é muitas vezes o número que importa.
- O **Go** ganha para serviços em rede e equipas: as goroutines fizeram o servidor
  mais rápido aqui, as compilações são quase instantâneas, o deployment é um único
  binário estático, e a linguagem é deliberadamente pequena o suficiente para uma
  equipa inteira a ler da mesma forma. Troca controlo de pico por produtividade e
  teve o melhor débito no teste de servidor desta academia.
- O **Rust** ganha onde os limites da máquina apertam: a menor memória, sem pausas de
  GC, e um compilador que torna classes inteiras de bugs — data races,
  use-after-free — *impossíveis* em vez de meramente detetáveis. Pagas em tempos de
  compilação e em aprender a satisfazer o borrow checker, e acima de certa contenção
  as suas primitivas mais simples precisam de mais cuidado (o resultado do mutex).

A meta-lição por baixo das três: **o runtime por baixo de uma primitiva define o seu
preço, e a única forma de saber o teu custo é medir a tua carga.** O mesmo `Mutex`, o
mesmo `INSERT`, o mesmo `for` significam coisas diferentes em linguagens diferentes,
e o método inteiro desta academia — constrói de três formas, verifica byte-idêntico,
depois mede — é a habilidade reutilizável. As linguagens foram o assunto; medir
trade-offs foi o objetivo.

## Cantinho da entrevista

<Interview question="Como escolherias entre Python, Go e Rust para um serviço novo?">

Identificando a restrição que aperta, depois casando-a com os trade-offs medidos de
cada linguagem. Se o gargalo é tempo de engenharia — exploração, trabalho de dados,
requisitos a mudar — a velocidade de escrita do Python ganha, e a sua penalização de
cálculo é irrelevante quando os hot paths são nativos ou o programa não é limitado
por débito. Se é um serviço em rede construído por uma equipa que precisa de andar
depressa e fazer deploy simples, a concorrência barata de goroutines do Go, as
compilações quase instantâneas, e o único binário estático são difíceis de bater —
produziu o melhor débito de servidor em testes que corri. Se as restrições que
apertam são footprint de memória, previsibilidade de latência (sem pausas de GC), ou
eliminar classes inteiras de bugs em tempo de compilação, o modelo sem-runtime e
sem-GC do Rust e o seu borrow checker pagam o seu custo de aprendizagem. Crucialmente
não responderia por reputação — nomearia a restrição, estimaria contra trade-offs
conhecidos (linguagens compiladas agrupam-se no cálculo, o CPython fica ~7× atrás, o
Rust segura a menor memória, o Go domina a concorrência fácil), e se a decisão for
renhida ou de alto risco, prototiparia o hot path em dois candidatos e mediria. As
linguagens são ferramentas com custos mensuráveis; a escolha profissional é casar
custo com restrição, não preferência.

</Interview>

<Interview question="Qual é a ideia mais transferível de comparar linguagens desta forma?">

Que o custo de uma primitiva é definido pelo runtime por baixo dela, não pelo seu
nome, por isso tens de medir a tua carga real em vez de raciocinar a partir de
rótulos. O mesmo `Mutex`, `INSERT`, ou loop `for` compila para maquinaria diferente
em linguagens diferentes — um park em userspace vs uma syscall futex, um parâmetro
ligado vs bytecode interpretado — e a única forma fiável de saber a diferença é
construir a mesma coisa de mais do que uma forma, verificar que os outputs são
idênticos, e cronometrá-los. Esse método — trabalho idêntico, verificado byte-a-byte,
depois medido — é mais valioso do que qualquer número individual que produz, porque
os números envelhecem e as cargas diferem mas a disciplina de ancorar uma decisão em
prova medida, nos teus inputs e no teu hardware, transfere-se para todas as questões
de performance que alguma vez enfrentarás. As linguagens foram o currículo; o hábito
de medir trade-offs é a habilidade.

</Interview>
