Everything this academy built converges here. Part 1 taught the languages;
Part 2, engineering discipline; Part 3, real tools; Part 4, concurrency and a
server; Part 5, the frameworks a web app is made of. The capstone is a single
small application — a **guestbook** — that composes those framework pieces into
one coherent pipeline, and then does the thing tutorials almost never do:
**attacks itself**, and proves it holds.

A guestbook is deceptively perfect for this. It is tiny — anyone leaves a name
and a message, everyone sees the list — yet it sits on top of the two most
exploited vulnerability classes in the history of the web: **SQL injection**
(the message goes into a database) and **cross-site scripting** (the message
comes back out onto a page). Get either wrong and a comment field becomes a
data-destroying or account-stealing hole. Get both right, from first principles,
and you understand the security spine of essentially every web application.

<Aside type="note" title="One principle, defended twice">
  Both defences are the same idea — *treat untrusted input as data, never as
  code, and make that the default*. On the way **in**, parameterized queries
  keep the message out of SQL syntax. On the way **out**, autoescaping keeps it
  out of HTML syntax. The capstone's whole thesis is that injection is one bug
  class with two façades, and one principle closes both doors.
</Aside>

## The pipeline

A submission flows through three stages, each a miniature of a Part 5 lesson:

<Mermaid title="validate -> store (parameterized) -> render (autoescaped)" code={`flowchart LR
  U["user submits {author, body}"] --> V["validate: accumulate every error"]

  V -->|clean| S["store: parameterized INSERT (? placeholders)"]

```python title="guestbook.py"
    errors = validate_submission(author, body)

        sql, params = insert_sql(author.strip(), body.strip())

def render_page(store: Store) -> str:

    lines = ['<ul class="guestbook">']
    for c in store.all():
        lines.append(

            f"{escape_html(c.body)}</li>"
        )
    lines.append("</ul>")
    return "\n".join(lines)
```

```rust title="src/lib.rs"
pub fn submit(store: &mut Store, author: &str, body: &str) -> Vec<String> {
    let errors = validate_submission(author, body);
    if errors.is_empty() {
        let (sql, params) = insert_sql(author.trim(), body.trim());
        store.execute_insert(&sql, &params); // values bound as data
    }
    errors
}

pub fn render_page(store: &Store) -> String {
    let mut html = String::from("<ul class=\"guestbook\">\n");
    for c in store.all() {
        html.push_str(&format!(
            "  <li><strong>{}</strong>: {}</li>\n",
            escape_html(&c.author),
            escape_html(&c.body)
        ));
    }
    html.push_str("</ul>");
    html
}
```

## The store: modelling what a database driver does

To prove the SQL-injection defence without a real database, the store is a
strict stand-in. It accepts **only** the exact parameterized insert and binds
the params as row data — exactly what a real driver does: the values are bound,
never executed as SQL.

<Tabs syncKey="lang">
<TabItem label="Python">
```python title="guestbook.py"
def execute_insert(self, sql: str, params: list[str]) -> None:
    if sql != "INSERT INTO comments (author, body) VALUES (?, ?)":
        raise ValueError("store only accepts the parameterized comment insert")
    if len(params) != 2:
        raise ValueError(f"expected two bound params, got {len(params)}")
    # Bind params as DATA — whatever is in them is stored verbatim, never SQL.
    self._comments.append(Comment(author=params[0], body=params[1]))
```
</TabItem>
<TabItem label="Go">
```go title="guestbook.go"

	if sql != "INSERT INTO comments (author, body) VALUES (?, ?)" {
		return fmt.Errorf("store only accepts the parameterized comment insert")
	}
```

<TabItem label="Rust">
```rust title="src/lib.rs"
pub fn execute_insert(&mut self, sql: &str, params: &[String]) {
    assert_eq!(
        sql, "INSERT INTO comments (author, body) VALUES (?, ?)",
        "store only accepts the parameterized comment insert"
    );
    assert_eq!(params.len(), 2, "expected two bound params");
    // Bind params as DATA. Whatever is in them — including a '; DROP...'
    // string — is stored verbatim as a value; it is never SQL.
    self.comments.push(Comment {
        author: params[0].clone(),
        body: params[1].clone(),
    });
}
```
</TabItem>
</Tabs>

This models the real guarantee precisely: the driver receives the SQL and the
values *separately*. The SQL is fixed template text with `?` holes; the values
fill the holes as data. A value of `'; DROP TABLE comments; --` is stored as the
literal characters of a comment body — the database never sees a second
statement to execute.

## Attack #1 — SQL injection, defeated

The adversarial test seeds a row, submits the classic drop-table payload as a
comment body, and asserts that the existing row survives and the payload was
stored as inert data:

```python title="test_guestbook.py"
def test_sql_injection_payload_is_stored_as_inert_data_table_survives(self):
    store = Store()
    submit(store, "Alice", "first comment")
    errors = submit(store, "Mallory", "'; DROP TABLE comments; --")
    self.assertEqual(errors, [])
    self.assertEqual(len(store.all()), 2)
    self.assertEqual(store.all()[0].body, "first comment")
    self.assertEqual(store.all()[1].body, "'; DROP TABLE comments; --")
```

```rust title="src/lib.rs (test)"
fn sql_injection_payload_is_stored_as_inert_data_table_survives() {
    let mut store = Store::new();
    submit(&mut store, "Alice", "first comment"); // an existing row
    // The attack: a body that tries to drop the table.
    let errors = submit(&mut store, "Mallory", "'; DROP TABLE comments; --");
    assert!(errors.is_empty());
    // The "existing row" is still there — nothing was dropped.
    assert_eq!(store.all().len(), 2);
    assert_eq!(store.all()[0].body, "first comment");
    // The payload was stored verbatim as a plain string value.
    assert_eq!(store.all()[1].body, "'; DROP TABLE comments; --");
}
```
</TabItem>
</Tabs>

The attack is accepted as an ordinary comment — and does nothing. Its dangerous
characters are just characters in a string field. Contrast the string-built SQL
this defends against, where `... VALUES ('Mallory', ''); DROP TABLE comments;
--')` would execute the drop. Parameterization is why the payload is data, not a
command.

## Attack #2 — XSS, defeated

The second adversarial test submits a `<script>` and asserts the rendered page
shows it as escaped text, with no live tag anywhere:

```rust title="the assertion, all three languages agree"
submit(&mut store, "Mallory", "<script>alert(document.cookie)</script>");
let page = render_page(&store);
assert!(page.contains("&lt;script&gt;alert(document.cookie)&lt;/script&gt;"));
assert!(!page.contains("<script>")); // no live script tag anywhere
```

The browser receiving this page renders the literal text `<script>…`, it does
not execute anything. Autoescaping turned the attack into inert content.

## The whole thing, byte-identical

Run the demo — two normal comments, then the two attacks — and all three
languages produce the exact same page:

```html
<ul class="guestbook">
  <li><strong>Ana</strong>: Love this academy!</li>
  <li><strong>Bruno</strong>: The Rust track is superb.</li>
  <li><strong>Mallory</strong>: &#39;; DROP TABLE comments; --</li>
  <li><strong>Eve</strong>: &lt;script&gt;alert(&#39;pwned&#39;)&lt;/script&gt;</li>
</ul>
```

Four comments stored — the SQL-injection attempt dropped nothing. Both hostile
payloads appear as harmless escaped text — the XSS executed nothing. The
end-to-end test asserts exactly this: three rows survive, and the page contains
the *escaped* forms of both attacks and no live `<script>`. Two attacks, two
defences, one principle, three languages — identical to the byte.

## Exercises

**1. Mount it on the real server (all three).** Wire the guestbook onto the
[Part 4 mini-NGINX](/systems-programming-academy/part-4-concurrency/mini-nginx/):
`GET /` renders the page, `POST /comment` runs `submit` and re-renders (or shows
the error list). Now it is a running web app you can open in a browser and try
to hack live. What did you have to add, and what was already handled?

<details>
<summary>Solution sketch</summary>

You add HTTP glue — routing `GET /` vs `POST /comment`, parsing the form body,
turning validation errors into a 400 with the error list — but the *security*
was already handled by the composed frameworks: nothing about serving it over
HTTP reopens the injection doors, because the defences live in the store and the
renderer, not in the transport. That separation — transport concerns vs
domain/security concerns — is the architecture lesson from Part 2 (hexagonal)
paying off: you extended the outside without touching the safe core.

</details>

**2. Add a second output format (any language).** Render the same comments as
JSON for an API (`GET /comments.json`). What is the equivalent of autoescaping
for JSON, and why can you not reuse the HTML escaper?

<details>
<summary>Solution sketch</summary>

JSON has its own escaping (quotes, backslashes, control characters) and its own
injection risk if you build it by string concatenation — the fix is to use a
real JSON encoder that escapes for the *JSON* grammar, exactly as HTML needs an
HTML escaper. Reusing the HTML escaper would both under-escape (miss `\` and
control chars) and over-escape (turn `<` into `&lt;` inside a JSON string, which
is wrong there). This is the context-dependence of escaping from the template
lesson, made concrete: every output grammar needs its own encoder.

</details>

**3. Store to real SQLite (Rust or Go).** Replace the in-memory store with an
actual database (`rusqlite` / `database/sql`), executing the same
`(sql, params)` pair against a real `comments` table. Submit the drop-table
payload and confirm — against a real SQL engine — that the table still exists
afterward. What changed in your code, and what deliberately did not?

<details>
<summary>Solution sketch</summary>

Only the store's internals change — it now calls the driver's prepared-statement
API with the same `(sql, params)` pair; the pipeline, the validation, the
rendering, and every test above are untouched. Running the drop-table payload
against real SQLite and finding the table intact is the proof that the
parameterized-insert guarantee was never a simulation — the in-memory store
modelled the real driver's contract faithfully, which is why swapping in the
real one requires no change to the security-relevant code. That is dependency
injection (Part 2 and 5) delivering its promise: the boundary was the seam, and
the seam held.

</details>

## Interview corner

<Interview question="Walk me through defending a comment feature against the two classic injection attacks.">

There are two boundaries and one principle. Going into storage, the attack is SQL
injection, and the defence is parameterized queries: the SQL is fixed text with
placeholders and the user's message is sent to the database separately, bound as
a value, so a `'; DROP TABLE ...` message is stored as literal data and never
executed. Coming out onto the page, the attack is cross-site scripting, and the
defence is autoescaping: every value is HTML-escaped by default before it reaches
the browser, so a `<script>` message renders as inert text rather than a live
tag. The unifying idea is *treat untrusted input as data, not code, and make that
the default* — and crucially, place each defence at its boundary (the database
layer and the rendering layer), so the application code that ties them together
cannot accidentally bypass either. Validation is a third, separate layer for
well-formedness; it is not where injection is stopped, and pretending otherwise —
"we block SQL keywords" — is a known-bad approach.

</Interview>

<Interview question="Why not just sanitize or reject dangerous-looking input at the front door instead of these layered defences?">

Because "dangerous-looking" is undefinable and the front door is the wrong place.
A legitimate comment might contain `DROP TABLE` (someone discussing databases) or
`<script>` (someone discussing HTML); rejecting them is both wrong and
insufficient, since attackers trivially evade keyword blocklists with encoding
and alternate syntax. Sanitizing — stripping or altering input — loses real data
and still leaves you guessing at every dangerous form. The robust approach makes
the input *harmless in context* rather than trying to recognize it: parameterize
so content cannot be SQL, escape so content cannot be HTML. The value is stored
and shown faithfully, yet powerless as code. This is defense by construction at
the boundary versus defense by pattern-matching at the perimeter, and the former
is why frameworks that default to it have made these bugs rare where they are
used.

</Interview>

<Interview question="This capstone uses an in-memory store. How do you know the SQL-injection defence is real and not an artifact of the fake store?">

Because the store models the real driver's contract exactly at the seam that
matters: it accepts a fixed SQL string plus a separate params list, and binds the
params as data — which is precisely the prepared-statement interface a real
driver exposes. The guarantee under test is "the user value travels as a bound
parameter, not as SQL text", and that is a property of the *call* (`insert_sql`
producing `?` placeholders and a params vector), not of the storage backend.
Swapping the in-memory store for real SQLite changes only the store's internals
and leaves the pipeline, the produced `(sql, params)`, and the tests unchanged —
which is itself the proof, and exercise 3 makes it concrete by running the
payload against a real database. The dependency-injection boundary (Part 2 and 5)
is what lets the fake and the real store be interchangeable without weakening the
guarantee — the seam was designed so the security-relevant behavior lives above
it, in the parameterized call, not below it in the backend.

</Interview>
