This academy taught three languages not in parallel but **against each other** —
the same program, built three ways, measured on the same machine. That structure
was the whole pedagogy: a trade-off you can *see in a number* is understood in a
way a paragraph can never deliver. This final reflection gathers those numbers
into one place and asks the question every engineer eventually faces: **which
language, and why?**

Every figure below was measured in an earlier lesson, on the same 20-core Linux
machine, and is linked to where it came from. Nothing here is a vibe; it is the
academy's own data, synthesized.

<Aside type="caution" title="Numbers are true, rankings are not">
  These measurements are honest for what they measured — specific workloads, one
  machine, one afternoon. They are useless as a universal ranking. "Rust is
  fastest" is the wrong lesson; "here is exactly what each choice costs on this
  kind of work, so measure yours" is the right one. Read the numbers as evidence
  for *how to think*, not as a leaderboard.
</Aside>

## Startup and footprint: what you pay to exist

Before doing any work, a program pays to start and to occupy memory. For a CLI
invoked thousands of times in a pipeline, or a sidecar you run hundreds of, this
is often the number that decides.

<Benchmark
  metric="Cold startup (empty input) — from the log-analyzer project (lower is better)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust (native binary)", value: 2, unit: "ms" },
    { label: "Go (native binary)", value: 3, unit: "ms" },
    { label: "Python (interpreter boot)", value: 26, unit: "ms" },
  ]}
/>

<Benchmark
  metric="Resident memory under load — from the mini-NGINX server (lower is better)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust", value: 2.3, unit: "MB" },
    { label: "Go", value: 10.5, unit: "MB" },
    { label: "Python", value: 22.2, unit: "MB" },
  ]}
/>

The shape repeats everywhere the academy measured memory (the
[exporter](/systems-programming-academy/part-3-devops/prometheus-exporter/)
recorded 2.0 / 7.6 / 21.3 MB for the same workload). Rust carries no runtime and
no garbage collector, so its floor is the OS and a few threads. Go's runtime and
GC cost a handful of megabytes but buy enormous convenience. Python's interpreter
is tens of megabytes before your code runs. None of this matters for a
long-running service where startup amortizes to nothing — and all of it decides a
function-as-a-service cold start or a memory-capped container.

## Throughput: the cost of the same work

The [log analyzer](/systems-programming-academy/part-3-devops/log-analyzer/)
parsed 500,000 lines of identical work, verified byte-identical across all three:

<Benchmark
  metric="Parse 500k log lines (lower is better)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust (release)", value: 98, unit: "ms" },
    { label: "Go (compiled)", value: 131, unit: "ms" },
    { label: "Python 3.12 (CPython)", value: 736, unit: "ms" },
  ]}
/>

Rust and Go land within ~30% of each other; Python is ~7× slower on this
CPU-bound text work. That ratio — compiled languages clustered, CPython roughly
an order of magnitude behind on raw compute — held everywhere the academy
measured pure computation. It is also exactly why Python's ecosystem pushes hot
loops into C (NumPy, the `re` engine, native extensions): idiomatic Python calls
fast native code rather than looping in bytecode.

## Concurrency: where the runtimes diverge most

The [threads lesson](/systems-programming-academy/part-4-concurrency/threads-shared-state/)
hammered one shared counter — four threads, a million increments each — and the
numbers told three different stories:

<Benchmark
  metric="4 threads × 1M increments of a shared counter (lower is better)"
  lowerIsBetter={true}
  rows={[
    { label: "Rust — mutex per increment", value: 356, unit: "ms" },
    { label: "Go — mutex per increment", value: 112, unit: "ms" },
    { label: "Go — atomic", value: 49, unit: "ms" },
    { label: "Rust — atomic", value: 51, unit: "ms" },
    { label: "batched (both, lock once)", value: 0.3, unit: "ms" },
  ]}
/>

Two findings shaped the whole concurrency arc. **Atomics cost the same in Rust
and Go (~50 ms)** because both compile to the same CPU instruction — at that
level the language is irrelevant, you are benchmarking cache coherence. But
**Go's contended mutex beat Rust's by 3×**, because Go's lock is integrated with
its userspace scheduler while Rust's `std::sync::Mutex` makes kernel futex calls
under brutal contention. Same primitive name, different machinery underneath — the
recurring lesson of Part 4.

And Python? The GIL means four CPU-bound threads run *slower* than one
([measured](/systems-programming-academy/part-4-concurrency/threads-shared-state/):
60 ms vs 54 ms). Python threads parallelize I/O, never CPU — which is not a bug
to fix but an architecture to design around (multiprocessing, or native code that
releases the GIL).

The [mini-NGINX server](/systems-programming-academy/part-4-concurrency/mini-nginx/)
put it all together under real HTTP load:

<Benchmark
  metric="20k HTTP requests, concurrency 50 (higher is better)"
  lowerIsBetter={false}
  rows={[
    { label: "Go (goroutine per connection)", value: 71143, unit: "req/s" },
    { label: "Rust (thread pool)", value: 53373, unit: "req/s" },
    { label: "Python (asyncio, 1 core)", value: 12948, unit: "req/s" },
  ]}
/>

Go's goroutine-per-connection model — cheap concurrency, multicore by default —
is purpose-built for servers and won throughput. Rust used a **fraction of the
memory** (2.3 vs 10.5 MB) but its shared-lock thread pool contended. Python's
single-core event loop serialized everything, the price of the GIL for CPU-ish
work. Three models, three shapes, each best somewhere.

## What the compiler buys, in one picture

Beyond numbers, the academy showed a qualitative axis: **when do you find out
you were wrong?**

<Mermaid title="When each language catches the classic concurrency bug" code={`flowchart TD
  BUG["data race: two threads mutate shared state unsynchronized"]
  BUG --> RUST["Rust: compile error (E0499) - the bug never exists"]
  BUG --> GO["Go: compiles & runs wrong; race detector finds it IF a test hits it"]
  BUG --> PY["Python: GIL narrows the window; still a latent bug, no detector"]`} />

The [threads lesson](/systems-programming-academy/part-4-concurrency/threads-shared-state/)
demonstrated this concretely: the same racy counter is a **compile error** in
Rust, silently loses 65% of its increments in Go (caught only by `go test
-race`), and is a quiet latent bug in Python. This is the deepest trade-off of
the three: Rust moves failure to compile time at the cost of fighting the borrow
checker; Go and Python defer it to runtime for faster authoring. Neither is free;
both are right for different stakes.

## So: which language?

The honest answer the academy earns is a set of **conditions**, not a winner.

<Mermaid title="A decision sketch, grounded in the measurements" code={`flowchart TD
  Q["what dominates your problem?"]
  Q -->|"iteration speed, glue, data/ML, scripts"| PY["Python: unbeatable authoring speed; push hot loops to native"]
  Q -->|"servers, concurrency, teams, simple deploys"| GO["Go: cheap concurrency, fast compile, one binary, easy to read"]
  Q -->|"memory/latency limits, safety-critical, systems"| RUST["Rust: no GC, no runtime, races impossible - at compile-time cost"]`} />

- **Python** wins when *your* time is the bottleneck, not the CPU's: exploration,
  data and ML, automation, glue. Its 7× compute penalty is irrelevant when the
  hot path is native code or the program runs once. It is the fastest language to
  *write*, which is often the number that matters.
- **Go** wins for networked services and teams: goroutines made the fastest server
  here, compiles are near-instant, deployment is a single static binary, and the
  language is deliberately small enough that a whole team reads it the same way.
  It trades peak control for productivity and got the best throughput in this
  academy's server test.
- **Rust** wins where the machine's limits bind: the smallest memory, no GC
  pauses, and a compiler that makes whole bug classes — data races, use-after-free
  — *impossible* rather than merely detectable. You pay in compile times and in
  learning to satisfy the borrow checker, and above a certain contention its
  simplest primitives need more care (the mutex result).

The meta-lesson underneath all three: **the runtime under a primitive sets its
price, and the only way to know your cost is to measure your workload.** The same
`Mutex`, the same `INSERT`, the same `for` loop mean different things in different
languages, and this academy's entire method — build it three ways, verify
byte-identical, then measure — is the reusable skill. The languages were the
subject; measuring trade-offs was the point.

## Interview corner

<Interview question="How would you choose between Python, Go, and Rust for a new service?">

By identifying the binding constraint, then matching it to each language's
measured trade-offs. If the bottleneck is engineering time — exploration, data
work, changing requirements — Python's authoring speed wins, and its compute
penalty is irrelevant when hot paths are native or the program is not
throughput-bound. If it is a networked service built by a team that needs to move
fast and deploy simply, Go's cheap goroutine concurrency, near-instant compiles,
and single static binary are hard to beat — it produced the best server
throughput in tests I have run. If the binding constraints are memory footprint,
latency predictability (no GC pauses), or eliminating whole bug classes at
compile time, Rust's no-runtime, no-GC model and its borrow checker pay for their
learning cost. Crucially I would not answer from reputation — I would name the
constraint, estimate against known trade-offs (compiled languages cluster on
compute, CPython trails ~7×, Rust holds the least memory, Go owns easy
concurrency), and if the decision is close or high-stakes, prototype the hot path
in two candidates and measure. The languages are tools with measurable costs;
professional choice is matching cost to constraint, not preference.

</Interview>

<Interview question="What is the single most transferable idea from comparing languages this way?">

That a primitive's cost is set by the runtime beneath it, not by its name, so you
must measure your actual workload rather than reason from labels. The same
`Mutex`, `INSERT`, or `for` loop compiles to different machinery in different
languages — a userspace park vs a futex syscall, a bound parameter vs interpreted
bytecode — and the only reliable way to know the difference is to build the same
thing more than one way, verify the outputs are identical, and time them. That
method — identical work, byte-verified, then measured — is more valuable than any
individual number it produces, because numbers age and workloads differ but the
discipline of grounding a decision in measured evidence, on your inputs and your
hardware, transfers to every performance question you will ever face. The
languages were the curriculum; the habit of measuring trade-offs is the skill.

</Interview>
